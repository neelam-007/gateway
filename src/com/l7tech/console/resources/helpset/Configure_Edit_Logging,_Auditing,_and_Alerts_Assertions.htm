Audit Assertion

The Audit Assertion allows you to set the threshold at which audit records are viewed in the Gateway Audit Events window (see View Audit Events). Auditing events are typically assertion, message processing, or boot messages and administrative events, such as object changes.

 

For example, when used in an "At least one assertion must evaluate to true" assertion folder (see Configure and Edit Policy Logic Assertions) after an Evaluate Request XPath assertion (see Configure and Edit Message Validation/Transformation Assertions), the Audit Assertion will only execute and audit the request message if the XPath assertion fails. When this happens, XPath query results are reported in the Gateway Audit Events window. If the XPath assertion in this scenario succeeds, then the Audit Assertion does not execute.

 

WARNING: The default "warning" level of events logged by the SecureSpan™ Gateway cannot be changed in the SecureSpan™ Manager. To override the default "warning" audit message threshold, you must change the applicable property in the SecureSpan Gateway's "/ssg/etc/conf/serverconfig_override.properties" file. See the "SecureSpan Gateway Installation and Configuration Manual" for more information. Changing the default log level below the "warning" level, such as to "info" or "finer," will yield a large number of audit messages. Before changing the log level, contact Layer 7 Technologies as outlined in Technical Support.

 

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The "Audit Assertion" assertion is added to the Policy Development window. Right-click the assertion. A drop-down menu appears. Select Audit Assertion Properties.

  1. The Audit Assertion Properties form appears with a default severity level of Warning that will include all audit alerts that may or may not require follow-up.

  1. Click [OK].

 

NOTE: To remove the auditing of requests, you must delete the Audit Assertion. To do so, right-click the assertion name in the Policy Development window. A drop-down menu appears. Select [Delete Assertion].

 

Send SNMP Trap

The Send SNMP Trap assertion allows you to instruct the SecureSpan™ Gateway to broadcast a Simple Network Management Protocol (SNMP) trap. When the Send SNMP Trap assertion is encountered in a policy execution path, an SNMP trap event will be broadcasted to a predefined network address. The assertion is typically used to trigger an alert based on the result of a previous assertion. For example, two assertions can be combined into a logical "One or More" folder (see Configure and Edit Policy Logic Assertions), the first assertion requiring validation, the later being the Send SNMP Trap assertion. If the assertion requiring validation fails, then the Send SNMP Trap assertion will execute, hence broadcasting the alert.

Add and Configure the SNMP Trap Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The SNMP Properties form appears. Enter the network address that should receive the SNMP alert into the SNMP Hostname field.

  2. The default SNMP trap destination port is set to "162." To configure a different port, select the Custom option and enter an alternate port number into the field.

NOTE: Port 162 is the IANA (Internet Assigned Numbers Authority) standard SNMP trap port.

  1. In the SNMP Community field, enter the SNMP community that should be used by the SNMP trap.

  2. In the Text to Send field, configure the contents of the SNMP trap.

  3. In the OID to Send field, complete the SNMP trap OID.

  4. Click [OK]. The assertion is added to the Policy Development window.

Edit the SNMP Trap Assertion

  • In the Policy Development window, right-click the "Send SNMP Trap" assertion. A drop-down menu appears. Select SNMP Trap Properties

  • The SNMP Properties form appears. Edit the assertion properties as outlined in Add and Configure the SNMP Trap Assertion.

 

Send Email Message

The Send Email Message assertion allows you to instruct the SecureSpan™ Gateway to deliver a pre-configured email message whenever the assertion is encountered in a policy.

 

The placement of the assertion in the policy path determines when and why an email is sent. For example, the assertion could be placed in an "At least one assertion must evaluate to true" assertion folder after an Evaluate Response XPath assertion (see Configure and Edit Message Validation/Transformation Assertions). If the required response message element is not found and the Evaluate Response XPath assertion fails, then the Send Email Message assertion will execute.

 

TIP: If the outgoing email account is improperly configured, then the Send Email Message assertion will fail. To configure the policy so that a failure of the Send Email Message assertion does not cause a total policy failure, place the assertion in an "At least one assertion must evaluate to true" folder with a Continue Processing assertion (see Configure and Edit Policy Logic Assertions). Since the pass or failure of the Send Email Message assertion is determined by the ability of the SecureSpan™ Gateway to generate and send an email message, destination server problems that prevent the receipt of the outgoing email will not cause the Send Email Message assertion to fail.  

 

Add and Configure the Send Email Message Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The Email Alert Properties form appears. If necessary, replace the default outgoing mail server name and port in the respective SMTP Server and Port fields with alternate values.  

  2. In the To field, enter the destination email address in the following format "mail@server.com."

  3. In the From field, optionally enter a response email address.

  4. In the Subject field, enter a descriptive name for the alert email.

  5. In the Message Body field, enter the content for the alert email.

  6. Click [OK]. The assertion is added to the Policy Development window.

Edit the Send Email Message Assertion

  • In the Policy Development window, right-click the "Send Email Alert" assertion. A drop-down menu appears. Select Email Alert Properties

  • The Email Alert Properties form appears. Edit the assertion properties as outlined in Add and Configure the Send Email Message Assertion.
