SecureSpan™ Manager Version Upgrade Issues

 TIP: See the Read Me file on the SecureSpan™ Manager Installation CD for specific information about the features and functions upgraded in version 3.0.

Upgrading version 2.1 of the SecureSpan™ Gateway and SecureSpan™ Manager to version 3.0 yields the following issues:

Security Policy Incompatibility

Version 2.1 policies will be automatically upgraded to 3.0-compatible policies the first time that they are accessed and saved in version 3.0 of the SecureSpan™ Manager (see Configure and Edit Security Policies for instructions). Since significant security policy assertion changes took place in version 3.0, however, some version 2.1 assertions have been modified or are no longer available:

SAML Security

 NOTE: For more information about the SAML Security assertion, see Configure and Edit Authentication Assertions.

The SAML Security assertion in the Authentication category of the [Assertions] tab changed in version 3.0. In version 2.1, the assertion was used to check timestamps on a SAML ticket. In 3.0, that functionality has expanded to also require a signed ticket. If your version 2.1 policies contain the SAML Security assertion, then:

 

  1. If not already present, load the target 2.1 security policy into the Policy Development window by one of the following methods:

 TIP: If not already disabled, it is strongly recommended that you disable a service while updating its security policy to prevent unsecured and unauthorized access. A disabled service contains a red "X" through its left-side icon. See Manage Published Services for information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

 NOTE: The version 2.1 [Web Services] tab has changed to [Services] in version 3.0 to include the display and management of XML, or non-SOAP, application security policies. See Services Overview for more information.

  1. The Policy Development window appears. If not present, enable the Policy Validation Messages window by selecting [View] > Policy Messages from the Main Menu (check box checked).

  2. In the Policy Development window, the now incompatible SAML Security assertion will display as an "Unknown assertion." To delete the assertion, right-click the "unknown assertion" listing in the Policy Development window. A drop-down menu appears. Select [Delete Assertion].

  1. A confirmation message appears. Click [Yes]. The assertion is removed from the Policy Development window. Re-add the 3.0 version of the SAML Security assertion, if required, as outlined in Configure and Edit Authentication Assertions and Configure and Edit Security Policies.

  2. In the Policy Tool Bar, click [Save]. The Manager automatically upgrades the remaining 2.1 assertions to 3.0-compatible assertions. Note any error or warning messages in the Policy Validation Messages window. If necessary, correct policy errors as outlined in the "Validate a Security Policy" section in Configure and Edit Security Policies.

  3. Repeat Steps #1-6 as required.

 

 NOTE: When upgrading a 2.1 security policy, ensure that you follow the policy and assertion information and rules in Security Policy Organization and Rules.

XML Message Signature and Encryption

 NOTE: For more information about the XML signature and encryption assertions, see Configure and Edit XML Security Assertions.

The XML signature and encryption assertions have changed dramatically in version 3.0 of the SecureSpan™ Manager. Called the XML Digital Signature assertion in the Authentication category and the Sign/Encrypt Request and Sign/Encrypt Response assertions in the XML Security category of the [Assertions] tab in version 2.1, these assertions are no longer available in version 3.0. Instead, full and partial XML message signature and encryption functionality has been separated and enhanced due to the required presence of one or more advanced authentication assertions.

 

During the conversion process, the Manager automatically includes all of the assertions required to replace and/or meet the new authentication pair requirements:

  • The version 2.1 XML Digital Signature authentication assertion no longer appears in version 3.0 of the Manager. If present in a 2.1 security policy, then the assertion will automatically be replaced with the new version 3.0 WSS Signature authentication assertion. See Configure and Edit Authentication Assertions for more information

  • Unlike version 2.1, the request message signing and response message encryption assertions require the presence of one or more authentication assertions. The Manager will automatically add authentication assertions to a converted version 2.1 policy as required to meet the version 3.0 requirements. See Configure and Edit XML Security Assertions for more information

  • In version 3.0, full and partial XML message signing and encryption is now separated into four different assertions in the XML Security category of the [Assertions] tab. Each assertion configures the signature or encryption requirement for a single message element. More than one assertion of a single type can be included in a security policy that requires the signature or encryption of more than one message element. During conversion, the Manager will automatically include all new assertions that are required to meet the XML signature and encryption requirements in the version 2.1 policy. See Configure and Edit XML Security Assertions for more information.

 

 NOTE: If the conversion yields assertions that you no longer require, then edit the security policy as outlined in Configure and Edit Security Policies.

 

Convert the XML signature and encryption assertions in a version 2.1 security policy to the version 3.0 assertions as follows:

 

  1. If not already present, load the target 2.1 security policy into the Policy Development window by one of the following methods:

 TIP: If not already disabled, it is strongly recommended that you disable a service while updating its security policy to prevent unsecured and unauthorized access. A disabled service contains a red "X" through its left-side icon. See Manage Published Services for information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

 NOTE: The version 2.1 [Web Services] tab has changed to [Services] in version 3.0 to include the display and management of XML, or non-SOAP, application security policies. See Services Overview for more information.

  1. The Policy Development window appears. If not present, enable the Policy Validation Messages window by selecting [View] > Policy Messages from the Main Menu (check box checked).

  2. In the Policy Tool Bar, click [Save]. The Manager automatically upgrades the 2.1 assertions to 3.0-compatible assertions. Note any error or warning messages in the Policy Validation Messages window. If necessary, correct policy errors as outlined in the "Validate a Security Policy" section in Configure and Edit Security Policies.

  1. Repeat Steps #1-3 in additional security policies as required.

 

 NOTE: When upgrading a 2.1 security policy, ensure that you follow the policy and assertion information and rules in Security Policy Organization and Rules.

Add-On Assertion Installation

 NOTE: For more information about the optional add-on assertions, see Optional Add-On Assertions Overview.

Upgrading version 2.1 of the SecureSpan™ Gateway to version 3.0 will require the re-installation and configuration of any optional add-on assertions in the version 2.1 security policies:

  • The SiteMinder™ authentication assertion from the Authentication category of the [Assertions] tab, called "SiteMinder Protected Resource" in the Policy Development window, installed as a separate assertion package

  • The Tivoli Access Manager (TAM) authentication assertion from the Authentication category of the [Assertions] tab, called "Tivoli Access Manager Assertion" in the Policy Development window, installed as a separate assertion package

  • The JMS routing assertion from the Message Routing category of the [Assertions] tab, called "Route to JMS Queue" in the Policy Development window, installed as JMS code.

If installed in the 3.0 version of the Gateway during the upgrade process, then converting the version 2.1 security policies in the SecureSpan™ Manager will automatically update any add-on assertions and, if applicable, configured JMS queues. If the add-on assertions were not installed in the Gateway during the upgrade process, however, then any add-on assertions will be identified as "Unknown assertions" in the version 3.0 Policy Development window. Install the required add-on assertions in the version 3.0 Gateway, then convert or re-convert the security policy in the Manager to enable the assertions:

 

 NOTE: For SiteMinder™ and TAM assertion installation and configuration information and instructions, see the SecureSpan™ Solution Add-On Assertion Installation Guide or contact your Administrator. For JMS assertion installation and configuration information and instructions, contact your Administrator.

 

  1. If not already present, load the target 2.1 security policy into the Policy Development window by one of the following methods:

 TIP: If not already disabled, it is strongly recommended that you disable a service while updating its security policy to prevent unsecured and unauthorized access. A disabled service contains a red "X" through its left-side icon. See Manage Published Services for information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

 NOTE: The version 2.1 [Web Services] tab has changed to [Services] in version 3.0 to include the display and management of XML, or non-SOAP, application security policies. See Services Overview for more information.

  1. The Policy Development window appears. If not present, enable the Policy Validation Messages window by selecting [View] > Policy Messages from the Main Menu (check box checked).

  2. In the Policy Tool Bar, click [Save]. The Manager automatically upgrades the 2.1 assertions to 3.0-compatible assertions. Note any error or warning messages in the Policy Validation Messages window. If necessary, correct policy errors as outlined in the "Validate a Security Policy" section in Configure and Edit Security Policies.

  1. Repeat Steps #1-3 in additional security policies as required.

 

 NOTE: When upgrading a 2.1 security policy, ensure that you follow the policy and assertion information and rules in Security Policy Organization and Rules.
