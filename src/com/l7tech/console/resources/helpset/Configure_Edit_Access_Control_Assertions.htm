Grant Access to Users/Groups

 NOTE: A policy with more than one Grant Access to Users/Groups assertion must organize the identities into an "At least one assertion must evaluate to true" assertion folder with its required authentication method assertion. To add a folder to the Policy Development window, right-click any assertion and select "Add 'One or More' Assertion" from the drop-down menu. Drag and drop a user or group into the folder as required. For more information, see Configure and Edit Policy Logic Assertions and Policy Organization.

The Grant Access to Users/Groups assertion allows you to provide service access to particular LDAP Identity Provider (LIP), Federated Identity Provider (FIP), and Internal Identity Provider (IIP) users and/or groups.

 

 WARNING: You cannot use the HTTP Digest authentication assertion (see HTTP Digest) with LDAP Identity Provider (LIP) users or groups (see Add and Edit LDAP Identity Providers).

 

  1. Add the assertion to the Policy Development window by one of three methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar

    • Right-click an "All assertions must evaluate to true" or "At least one assertion must evaluate to true" assertion folder. A drop-down menu appears. Select Add User or Group.

  1. The Search Identity Provider form appears. Select the LIP, FIP, or IIP name that contains the target user and/or group from the Search drop-down list.

 NOTE: To initiate a new search in the Search Identity Provider form, click [New Search]. To stop a search that is in progress, click [Stop].

  1. Select the search target users, groups, or both from the Type drop-down list.

  2. If necessary, specify any search criteria using the Name drop-down list and field.

 TIP: The Search Identity Provider form supports the following wild card characters in the Name field:

  • The asterisk truncation operator (*) retrieves terms with a common initial spelling. It substitutes a string of zero or more characters when searching the Internal Identity Provider, Federated Identity Providers, and LDAP Identity Providers

  •  The question mark wild card operator (?) substitutes zero or one character in a term. Use within a word or at the end of a word when searching the Internal Identity Provider.

  1. Click [Search]. Search results appear in the Search Results window. Select one or more users and/or groups as follows:

 NOTE: A single Grant Access to Users/Groups assertion will be added to the policy for each user and/or group selected in the Search Identity Provider form.

Select a Single User or Group

To select a single user or group for the policy, click the user or group's name and click [Select]. The Search Identity Provider form closes and an assertion for the user or group is added to the Policy Development window.

Select Multiple Users and/or Groups

Select multiple users or groups for the policy by one of two methods:

    • To select a consecutive range of users or groups, hold down your left mouse button as you move from the first user or group to the last user or group in your target range. Or, press [Shift] on your keyboard, left-click the first user or group in the range, and then left-click the last user or group in the range. Click [Select]

    • To add users or groups placed throughout the list, press [Ctrl] on your keyboard as you click each user or group name that you want to add. Click [Select].

HTTP Basic

 TIP: Since the HTTP Basic assertion requires the insertion of the password into the service message, it is recommended that you also select the SSL Transport assertion (see Configure and Edit Transport Layer Security (TLS) Assertions) when using basic HTTP authentication.

The HTTP Basic assertion allows you to require basic HTTP authentication user name, plain text password, and the authentication realm as a string in the Web service or XML application request headers. This assertion is a credential source that saves the user name and password from the HTTP headers for later authentication and authorization via the Grant Access to Users/Groups assertion (see Grant Access to Users/Groups).

 

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

 

HTTP Digest

 WARNING: You cannot use the HTTP Digest assertion with the SOAP Request with Attachment assertion (see Configure and Edit Message Validation/Transformation Assertions)or with LDAP Identity Provider (LIP) users or groups (see Add and Edit LDAP Identity Providers).

The HTTP Digest assertion allows you to require an HTTP user name, password, and authentication realm in the request headers of a "SHA" digest form for the Web service or XML application. This assertion is a credential source that saves the user name and digested password from the HTTP headers for later authentication and authorization via the Grant Access to Users/Groups assertion (see Grant Access to Users/Groups).

 

  • Add the assertion to the Policy Development window by one of two methods:

SSL with Client Certificate Authentication

 NOTE: The SSL with Client Certificate Authentication assertion is the same as the SSL or TLS Transport assertion (when client certificate authentication is enabled) from the Transport Layer Security (TLS) category in the [Assertions] tab (see Configure and Edit the Transport Layer Security (TLS) Assertion).

The SSL with Client Certificate Authentication assertion allows you to specify the SSL or TLS requirement for a policy. When SSL or TLS transport is required, then a client certificate requirement can also be set for the policy. When selected, the assertion will act like a credential source that saves the client certificate from the SSL-TLS handshake for later authentication and authorization via the Grant Access to Users/Groups assertion (see Grant Access to Users/Groups).

 

Add and Configure the SSL with Client Certificate Authentication Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The assertion appears in the Policy Development window with a default setting of "required." To optionally change the assertion setting, right-click the "Require SSL or TLS Transport with Client Certificate Authentication" assertion. A drop-down menu appears. Select SSL Properties.

  1. The SSL or TLS Options form appears. Select one of the following SSL or TLS requirement options:

    • Select the Required option to disallow requests that do not arrive over an SSL-secured connection. This is the default setting for the assertion. When selected, the client certificate authentication option is enabled. To require client certificate authentication of request messages in the policy, check the Require Client Certificate Authentication check box

    • Select the Optional option to configure the SecureSpan Gateway to match the behavior of the incoming request. Requests are not required to arrive over an SSL-secured connection, but will be processed anyway if they happen to do so

    • Select the Forbidden option to disallow requests that arrive over an SSL-secured connection. The assertion setting could be used to discourage users of a free service from consuming server SSL resources without paying for an upgraded account.

  1. Click [OK]. The modified assertion is added to the Policy Development window.

WSS UsernameToken Basic

 TIP: Since the WSS UsernameToken Basic assertion requires the insertion of the plain text password into the service message, it is recommended that you also select the SSL or TLS Transport assertion (see Configure and Edit Transport Layer Security (TLS) Assertions) when using basic WS Token authentication. The "WSS" in the WSS UsernameToken Basic assertion stands for "WS-Security." The assertion supports version 1.0 of the WS-Security standard. The SecureSpan™ Gateway creates and uses X.509 v3 certificates.

The WSS UsernameToken Basic assertion allows you to require basic WS usernametoken authentication user name, plain text password, and the authentication realm in a section of the XML message for the Web service or XML application. This assertion is a credential source that saves the user name and password from the WSS usernametoken for later authentication and authorization via the Grant Access to Users/Groups assertion (see Grant Access to Users/Groups).

WSS Signature

 NOTE: The "WSS" in the WSS Signature assertion stands for "WS-Security." The assertion supports version 1.0 of the WS-Security standard. The SecureSpan™ Gateway creates and uses X.509 v3 certificates.

The WSS Signature assertion allows you to require that the Web service or XML application request message:

  • Include an X.509 BinarySecurityToken containing a client certificate

  • Have at least one element signed by that client certificate's private key as a proof of possession of the private key for the client certificate.

 

The WSS Signature assertion is a credential source that saves the certificate from the X.509 BinarySecurityToken for later authorization via the Grant Access to Users/Groups assertion (see Grant Access to Users/Groups).

 

The WSS Signature assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab (see Configure and Edit XML Security Assertions).

 

 WARNING: The WSS Signature assertion, by itself, does not require that the request message contain a timestamp, and does not check the validity of any timestamp that might be present. To protect against stale or replayed messages, use the WSS Signature assertion  with the WSS Replay Protection assertion (see Configure and Edit XML Security Assertions).

  • The assertion is added to the Policy Development window.

 

 OPTION: The recipient of the WSS Signature assertion can be changed using the Change WSS Recipient form (see Change the WSS Assertion Recipient).

 

WS Secure Conversation

 NOTE: The WS Secure Conversation assertion supports the WS-Security 1.0 standard.

The WS Secure Conversation assertion allows you to require that request and response messages be secured using a secure conversation as opposed to session-less security. Specifically, messages must:

  • Include a "SecurityContextToken" referencing an already-established WS-SecureConversation session

  • Include at least one element signed with the shared secret from this session as proof of possession of the session shared secret

  • Be delivered to the same SecureSpan™ Gateway node on which the session is established. In practice, this typically requires that the request message arrive via HTTP(S) routing rather than JMS routing (see Configure and Edit Message Routing Assertions).

 

The WS Secure Conversation assertion is a credential source that saves the user that owns the session for later authorization via the Grant Access to Users/Groups assertion (see Grant Access to Users/Groups). The WS Secure Conversation assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab (see Configure and Edit XML Security Assertions).

 

 NOTE: The WS Secure Conversation assertion, by itself, does not require that the request message contain a timestamp, and does not check the validity of any timestamp that might be present. To protect against stale or replayed messages, use the WS Secure Conversation assertion with the WSS Replay Protection assertion (see Configure and Edit XML Security Assertions).

 

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

 

SAML Assertion

 WARNING: To avoid constraint conflicts, only a single SAML Assertion should be present in a policy. The SAML Assertion supports the SAML 1.1 standard.

The SAML Assertion allows you to require SAML constraints in a policy. SAML (Security Assertions Markup Language) validates a ticket to ensure that it falls within the required constraints. If validation succeeds, then the SecureSpan™ Gateway passes the message through to the service. If validation fails, then the SecureSpan Gateway returns a SOAP fault.

 

The SAML Assertion is a credential source that saves subject information for later authorization via the Grant Access to Users/Groups assertion (see Grant Access to Users/Groups). The SAML Assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab (see Configure and Edit XML Security Assertions). The SAML Assertion is also used as a credential source for an identity bridging configuration (see Identity Bridging).

Holder-of-Key

Check the Holder-of-Key check box to allow SAML Assertions that use the Holder-of-Key subject confirmation method (with the standard URI "urn:oasis:names:tc:SAML:1.0:cm:holder-of-key"). For such assertions, the SecureSpan™ Gateway will require that the subject demonstrate possession of the private key corresponding to the public key in the Subject certificate.  

 

The Holder-of-Key subject confirmation method currently requires that the request ticket's "SubjectConfirmation" element contain a "KeyInfo" element that contains a complete copy of the Subject's X.509 certificate. Any other form of Holder-of-Key ticket will be rejected by the SecureSpan Gateway.

 

The request Subject may use one of two methods to prove that they hold this key:

      • The request includes at least one element covered by a valid WSS message signature (see WSS Signature), and the signing certificate is the Subject certificate. Or,

      • The request arrived over SSL/TLS with client certificate authentication (see SSL with Client Certificate Authentication), and the client certificate exactly matches the Subject certificate.

 

When the Holder-of-Key subject confirmation method is selected, you can optionally check the Require Message Signature check box to require proof-of-possession using a WSS message signature. If the Require Message Signature check box is not selected, then the policy will require the SSL or TLS Transport assertion from the Transport Layer Security (TLS) category of the [Assertions] tab (see Configure and Edit the Transport Layer Security (TLS) Assertion).

Sender Vouches

 NOTE: The Sender Vouches subject confirmation method will typically only be used in a SAML identity bridging policy (see Identity Bridging).

Check the Sender Vouches check box to allow SAML Assertions that use the Sender Vouches subject confirmation method (with the standard URI "urn:oasis:names:tc:SAML:1.0:cm:sender-vouches"). For such assertions, the SecureSpan Gateway will require that the sender, presumably different from the subject, vouches for the verification of the subject.

 

Three conditions must be met in order to use the Sender Vouches subject confirmation method:

      1. An existing trust relationship with the sender, or "Attesting Entity", must be configured in the SecureSpan Gateway. To do so, import the sender's certificate, configured as a "SAML Attesting Entity" certificate, into the Trust Store (see Manage Certificates).

      2. The SAML ticket used by the SAML Assertion must be bound to the request message by one of the following methods:

        • Send the request over SSL using the sender certificate as the SSL client certificate, or

        • If SSL is not used, then the SAML ticket needs to be bound to the message with a WSS signature (see WSS Signature). One complication here is that the SAML ticket does not necessarily contain or refer to the sender certificate; it usually contains or refers to the subject certificate and, assuming that the ticket is signed, contains or refers to the certificate of the ticket issuer. In this method, therefore, the WSS signature must cover both the SAML Assertion and the relevant portions of the rest of the message that use the sender certificate as the signing certificate.  

      1. The format of the request message must conform with the OASIS Web Services Security: SAML Token Profile 1.0 standard. The SecureSpan Gateway does not support references to SAML Assertions that are not included with the request message.

 

 FYI: The OASIS Web Services Security: SAML Token Profile 1.0 standards document is available online: http://www.oasis-open.org/committees/download.php/1048/WSS-SAML-06.pdf.

 

If selected, optionally check the Require Message Signature check box to require proof-of-possession using an SSL client certificate. If the Require Message Signature check box is not selected, then the policy will require the SSL or TLS Transport assertion from the Transport Layer Security (TLS) category of the [Assertions] tab (see Configure and Edit the Transport Layer Security (TLS) Assertion)

Bearer

Check the Bearer check box to allow SAML Assertions that use the Bearer Token subject confirmation method (with the standard URI "urn:oasis:names:tc:SAML:1.0:cm:bearer"). Like an HTTP cookies, such assertions will always be assumed to belong to whatever message contains them, and the subject will be assumed to be the sender of the message.

 

 TIP: The Bearer Token subject confirmation method does not protect against an attacker modifying the message or stealing a copy of the assertion and attaching it to an unauthorized message. To protect the secrecy of the SAML Assertion when using the Bearer subject confirmation method, check the SSL-TLS Client Certificate Authentication check box discussed in Step # 2 above.

 

None

Check the None check box to allow SAML Assertions that do not contain a subject confirmation method.

 

 TIP: Not having a subject confirmation method exposes the system to various threats. To protect the secrecy of the SAML Assertion when a confirmation method is not used, check the SSL-TLS Client Certificate Authentication check box discussed in Step # 2 above.

One complication here is that the SAML ticket does not necessarily contain or refer to the sender certificate; it usually contains or refers to the subject certificate and, assuming that the ticket is signed, contains or refers to the certificate of the ticket issuer. In this method, therefore, the WSS signature must cover both the SAML Assertion and the relevant portions of the rest of the message that use the sender certificate as the signing certificate.  

      1. The format of the request message must conform with the OASIS Web Services Security: SAML Token Profile 1.0 standard. The SecureSpan Gateway does not support references to SAML Assertions that are not included with the request message.

 

 FYI: The OASIS Web Services Security: SAML Token Profile 1.0 standards document is available online: http://www.oasis-open.org/committees/download.php/1048/WSS-SAML-06.pdf.

WS-Trust Credential Exchange

 WARNING: The WS-Trust Credential Exchange assertion will be invalidated if the routing assertion in the policy is set to remove processed Security headers. When using the WS-Trust Credential Exchange assertion, you must configure the HTTP(S) Routing or SecureSpan Bridge Routing assertion to maintain the Security header in the message. To do so, select the "Leave current Security header in request before routing" option in the HTTP(S) Routing Properties form that is used by both assertions. See Configure and Edit Message Routing Assertions for more information. If the credentials in a message are covered by an XML Signature (with the Sign Request Element assertion discussed in Configure and Edit XML Security Assertions), then the signature will be invalidated when the credentials are replaced by the WS-Trust Credential Exchange assertion.

The WS-Trust Credential Exchange assertion takes credentials gathered by a preceding credential source assertion, such as the transport-level HTTP Basic assertion (see HTTP Basic) or message-level WSS UsernameToken Basic assertion (see WSS UsernameToken Basic), and sends them via a WS-Trust RequestSecurityToken (RST) SOAP request to a WS-Trust Security Token Service (STS). If the resulting SOAP response is a RequestSecurityTokenResponse (RSTR) and not a fault, and its RequestedSecurityToken element contains a valid security token (either a SAML Assertion or a UsernameToken) the assertion will replace the current request's credentials with that token. If the message's original credentials were XML-based, then the XML element containing those credentials will be removed from the message and replaced with the RequestedSecurityToken element.

XPath Credentials

 NOTE: In order to use the XPath Credentials assertion, both a user name and password must be configured for the identity or identities (see Grant Access to Users/Groups) in the policy.

The XPath Credentials assertion looks for a login (user name) and password in the current request using a pair of XPath expressions. If the target credentials are found in the message, then the SecureSpan™ Gateway sets the current request's credentials using the contents of the elements described by the XPath expressions and optionally removes the original elements from the request.

About the Namespace Map

The SecureSpan™ Manager cannot predict which namespace prefixes, if any, might be found in the messages that will be received by the SecureSpan Gateway. To allow the use of namespace prefixes in XPath expressions, the SecureSpan Gateway supports a user-editable namespace map attached to any XPath-based assertion (the XPath Credentials, Evaluate Request XPath, and Evaluate Response XPath (see Configure and Edit Message Validation/Transformation Assertions) assertions). Without namespace prefixes, constructing correct and namespace-aware XPath expressions is extremely complicated. For example, a correct, namespace-aware XPath expression that will match the top-level Envelope element in a SOAP 1.1 message without using prefixes is:

 

/*[local-name()="Envelope" and namespace-uri()="http://schemas.xmlsoap.org/soap/envelope"]

 

Since the http://schemas.xmlsoap.org/soap/envelope namespace URI is typically declared in the namespace map under the prefix "soapenv", the same XPath expression can be rewritten just as correctly as:

 

/soapenv:Envelope

 

To allow a given prefix to be used in the XPath expression in an XPath-based assertion, an entry for the prefix must be present in the assertion's namespace map. Using the custom mapping feature, a namespace map can be defined and saved in the XPath Credentials assertion. New namespaces can be added and removed from the default namespace list, but the default namespaces themselves cannot be removed or changed.

 

Namespace prefixes that may appear in messages received by the SecureSpan Gateway are unrelated to those found in the namespace map of XPath-based assertions, even if they may happen to be identical. In order for an XPath expression using prefixes to match namespace-qualified nodes in a message, the message's namespace URIs must exactly match those found in the assertion's namespace map. For example, consider the following simplified SOAP 1.1 message:

 

<Envelope xmlns="http://schemas.xmlsoap.org/soap/envelope">

  <Body>

    <someApplication:operationName xmlns:someApplication="urn:example.com:someApp"/>

  </Body>

</Envelope>

 

Even though this message's Body element appears without a namespace prefix, it is still qualified with the standard SOAP 1.1 namespace URI. The naive XPath expression "//Body" might match this particular message, but would fail to match another message that happened to employ a prefix for the SOAP 1.1 namespace URI. With the namespace map feature, the expression "//soapenv:Body" can match any message that uses the correct namespace URI, regardless of what prefix, if any, it uses to declare it.

SAML Browser Artifact

The SAML Browser Artifact assertion authenticates an HTTP GET request that has HTTP Basic credentials (see HTTP Basic) against an identity provider SAML endpoint. If authentication succeeds, the SecureSpan™ Gateway parses the redirect header and saves the "SAMLart" parameter in memory for future assertions in the same policy to use.

 

The SAML Browser Artifact assertion is useful for "mixed-mode" SAML interactions in which an initial request containing a user's credentials establishes a Single Sign-On session that can be used in subsequent browser-based requests from the same user.

 

 TIP: The saved SAML artifact value can be used in the Regular Expression assertion (see Configure and Edit Message Validation/Transformation Assertions) by entering the variable "${samlBrowserArtifact.artifact}" in the Replacement field in the Regular Expression Assertion form.

 

Add and Configure the SAML Browser Artifact Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The Configure SAML Browser/Artifact form appears. In the SAML Single Sign-On Endpoint URL field, enter the URL of a SAML identity provider endpoint that will accept an HTTP GET request with Basic credentials. The URL must include both the Single Sign-On (SSO) endpoint and the corresponding service endpoint. If the SSO system returns a 302 status code after processing, then the assertion succeeds and will proceed to process the service endpoint URL. If the SSO system returns a non-302 status code, if the target redirect URL cannot be parsed, or if the result does not match the SAML Artifact Query Parameter specified below, then the assertion will fail, possibly resulting in a failure of the policy.

 NOTE: The URL specified in the SAML Single Sign-On Endpoint URL field is only for authentication, not message routing. Even if the URL is the same as the endpoint of the service, the HTTP(S) Routing assertion (see Configure and Edit Message Routing Assertions) is still required to route service messages.

  1. The SAML Artifact Query Parameter field is populated with the default "SAMLart" value. SAMLart is the type of cookie returned from the SSO system, which is then sent on to the target URL. If the return cookie is not the same type as the value in the SAML Artifact Query Parameter, then the assertion will fail.

 NOTE: The SAMLart parameter should suffice for most usage scenarios of the SAML Browser Artifact assertion. The default value should only be changed if the administrator of the Single Sign-On system has chosen a different parameter name. The value in the SAML Artifact Query Parameter field is case sensitive.

  1. Click [OK]. The assertion is added to the Policy Development window.
