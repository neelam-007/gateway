Grant Access to Users/Groups

 NOTE: In the SecureSpan" Manager, the term "identity" includes both users and groups, and the term "user" represents an individual human or machine. If an existing policy contains a user or group that has since been removed from its corresponding identity provider, then the SecureSpan Manager will fail the policy during the final validation process. See the "Validate a Policy" section in Configure and Edit Policies for more information.

The Grant Access to Users/Groups assertion allows you to provide service access to particular LDAP Identity Provider (LIP), Federated Identity Provider (FIP), and Internal Identity Provider (IIP) users and/or groups.

 

 NOTE: You cannot use the HTTP Digest authentication assertion with LDAP Identity Provider (LIP) users or groups. For more information about the HTTP Digest assertion, see HTTP Digest. For more information about LIPs, see Add and Edit LDAP Identity Providers.

 

  1. Add the assertion to the Policy Development window by one of three methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar

    • Right-click an "All assertions must evaluate to true" or "At least one assertion must evaluate to true" assertion folder. A drop-down menu appears. Select Add User or Group.

  1. The Search Identity Provider form appears. Select the LIP, FIP, or IIP name that contains the target user and/or group from the Search drop-down list.

 NOTE: To initiate a new search in the Search Identity Provider form, click [New Search]. To stop a search that is in progress, click [Stop].

  1. Select the search target users, groups, or both from the Type drop-down list.

  2. If necessary, specify any search criteria using the Name drop-down list and field.

 TIP: The Search Identity Provider form supports the following wild card characters in the Name field:

"  The asterisk truncation operator (*) retrieves terms with a common initial spelling. It substitutes a string of zero or more characters when searching the Internal Identity Provider, Federated Identity Providers, and LDAP Identity Providers

"  The question mark wild card operator (?) substitutes zero or one character in a term. Use within a word or at the end of a word when searching the Internal Identity Provider.

Drag and drop a user or group into the folder as required. For more information, see Policy Organization and Rules and Configure and Edit Policies.

Select a Single User or Group

To select a single user or group for the policy, click the user or group's name and click [Select]. The Search Identity Provider form closes and an assertion for the user or group is added to the Policy Development window.

To switch back to the Policy Development window, click the [Policy View] button.

 

HTTP Basic

 NOTE: Since the HTTP Basic assertion requires the insertion of the password into the service message, it is recommended that you also select the SSL Transport assertion when using basic HTTP authentication. The HTTP Basic assertion allows you to require basic HTTP authentication user name, plain text password, and the authentication realm as a string in the Web service or XML application request headers. WSS Signature

 NOTE: The "WSS" in the WSS Signature assertion stands for "WS-Security." The assertion supports version 1.0 of the WS-Security standard. The SecureSpan" Gateway creates and uses X.509 v3 certificates.

The WSS Signature assertion allows you to require that the Web service or XML application request message:

  • Include an X.509 BinarySecurityToken containing a client certificate

  • Have at least one element signed by that client certificate's private key as a proof of possession of the private key for the client certificate.

 

The WSS Signature assertion is a credential source that saves the certificate from the X.509 BinarySecurityToken for later authorization via the Grant Access to Users/Groups assertion. For information on the Grant Access to Users/Groups assertion, see Grant Access to Users/Groups.

 

The WSS Signature assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab. See Configure and Edit XML Security Assertions for more information.

 

 NOTE: The WSS Signature assertion, by itself, does not require that the request message contain a timestamp, and does not check the validity of any timestamp that might be present. To protect against stale or replayed messages, use the WSS Signature assertion with the WSS Replay Protection assertion. See Configure and Edit XML Security Assertions for more information about the WSS Replay Protection assertion.

WS Secure Conversation

 NOTE: The WS Secure Conversation assertion supports the WS-Security 1.0 standard.

The WS Secure Conversation assertion allows you to require that request and response messages be secured using a secure conversation as opposed to session-less security. Specifically, messages must:

  • Include a "SecurityContextToken" referencing an already-established WS-SecureConversation session

  • Include at least one element signed with the shared secret from this session as proof of possession of the session shared secret

  • Be delivered to the same SecureSpan" Gateway node on which the session is established. In practice, this typically requires that the request message arrive via HTTP(S) routing rather than JMS routing. For more information about the message routing assertions, see Configure and Edit Message Routing Assertions.

 

The WS Secure Conversation assertion is a credential source that saves the user that owns the session for later authorization via the Grant Access to Users/Groups assertion. For information on the Grant Access to Users/Groups assertion, see Grant Access to Users/Groups. The WS Secure Conversation assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab. See Configure and Edit XML Security Assertions for more information.

 

 NOTE: The WS Secure Conversation assertion, by itself, does not require that the request message contain a timestamp, and does not check the validity of any timestamp that might be present. To protect against stale or replayed messages, use the WS Secure Conversation assertion with the WSS Replay Protection assertion. See Configure and Edit XML Security Assertions for more information about the WSS Replay Protection assertion.

SAML Assertion

 NOTE: The SAML Assertion supports the SAML 1.1 standard. To avoid constraint conflicts, only a single SAML Assertion should be present in a policy.

The SAML Assertion allows you to require SAML constraints in a policy. SAML (Security Assertions Markup Language) validates a ticket to ensure that it falls within the required constraints. If validation succeeds, then the SecureSpan" Gateway passes the message through to the service. If validation fails, then the SecureSpan Gateway returns a SOAP fault.

 

The SAML Assertion is a credential source that saves subject information for later authorization via the Grant Access to Users/Groups assertion (see Grant Access to Users/Groups). The SAML Assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab (see Configure and Edit XML Security Assertions). The SAML Assertion is also used as a credential source for an identity bridging-configuration. See Identity Bridging Overview for more information.

WS-Trust Credential Exchange

 NOTE: The WS-Trust Credential Exchange assertion will be invalidated if the routing assertion in the policy is set to remove processed Security headers. When using the WS-Trust Credential Exchange assertion, you must configure the HTTP(S) Routing or SecureSpan Bridge Routing assertion to maintain the Security header in the message. To do so, select the "Leave current Security header in request before routing" option in the HTTP(S) Routing Properties form that is used by both assertions. See Configure and Edit Message Routing Assertions for more information.

The WS-Trust Credential Exchange assertion takes credentials gathered by a preceding credential source assertion, such as the transport-level HTTP Basic assertion or message-level WSS UsernameToken Basic assertion, and sends them via a WS-Trust RequestSecurityToken (RST) SOAP request to a WS-Trust Security Token Service (STS). If the resulting SOAP response is a RequestSecurityTokenResponse (RSTR) and not a fault, and its RequestedSecurityToken element contains a valid security token (either a SAML Assertion or a UsernameToken) the assertion will replace the current request's credentials with that token. If the message's original credentials were XML-based, then the XML element containing those credentials will be removed from the message and replaced with the RequestedSecurityToken element.

 

 NOTE: If the credentials in a message are covered by an XML Signature, then the signature will be invalidated when the credentials are replaced by the WS-Trust Credential Exchange assertion. For more information about the Sign Request Element assertion, see Configure and Edit XML Security Assertions.

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The Configure WS-Trust Credential Exchange form appears. Enter the URL of the WS-Trust Security Token Service (STS) into the Token Service URL field. The STS must be running and configured to accept RequestSecurityToken (RST) requests containing the values configured below.

  2. In the wsp:AppliesTo: URI field, enter a URI that describes the service for which the token will be used. For example, "urn:example.com:services:echo" or "http://services.example.com/EchoService."

XPath Credentials

 NOTE: In order to use the XPath Credentials assertion, both a user name and password must be configured for the identity or identities in the service.

The XPath Credentials assertion looks for a login (user name) and password in the current request using a pair of XPath expressions. If the target credentials are found in the message, then the SecureSpan™ Gateway sets the current request's credentials using the contents of the elements described by the XPath expressions and optionally removes the original elements from the request.

 

Add and Configure the XPath Credential Source Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The Configure XPath Credential Source form appears. In the Login XPath Expression field, enter an XPath 1.0 expression that will locate the element containing the login ID or user name. For example, "//tns:credentials/tns:login."

 TIP: Before constructing XPath expressions for the login/user name and password elements, consult the service's namespace map to view and choose the appropriate namespace prefixes. To access the map, click [Namespaces]. The Edit Namespaces and Prefixes form appears with the default WSDL namespaces and prefixes.

  1. Optionally check the Remove Login from request if found check box. When selected, the SecureSpan™ Gateway will remove the element containing the login/user name value from the request message. Use this option when credentials must be authenticated by the SecureSpan Gateway but not communicated to the protected service.

  2. In the Password XPath Expression field, enter an XPath 1.0 expression that will locate the element containing the password. For example, "//tns:credentials/tns:password."  

  3. Optionally check the Remove Password from request if found check box. When selected, the SecureSpan Gateway will remove the element containing the password value from the request message.

 NOTE: The SecureSpan™ Gateway removes the credentials from the request message but saves the credentials in memory.

  1. Optionally add namespaces to the default namespaces gleaned from the Web service WSDL as follows:

 NOTE: You cannot change or remove the namespaces originating from the WSDL document. You can, however, add and remove any namespaces added with the [Namespaces] button in the Configure XPath Credential Source form.

    1. Click [Namespaces]. The Edit Namespaces and Prefixes form appears with the default WSDL namespaces and prefixes.

    2. Click [Add]. The XML Namespace form appears.

    3. Enter the prefix for the new namespace into the Prefix field.

    4. Enter the new namespace into the Namespace URL field.

    5. Repeat to enter additional namespaces. When completed, click [OK].

 NOTE: To remove a namespace from the Edit Namespaces and Prefixes form, select the target namespace from the table and click [Remove].

  1. In the Configure XPath Credential Source form, click [OK]. The assertion is added to the Policy Development window.

  • In the Policy Development window, right-click the "XPath credentials..." assertion. A drop-down menu appears. Select View/Edit XPath Credential Source Properties

  • The Configure XPath Credential Source form appears. Edit the assertion as required. See Add and Configure the XPath Credential Source Assertion for instructions.

 

SAML Browser Artifact

The SAML Browser Artifact assertion authenticates an HTTP GET request that has HTTP Basic credentials against an identity provider SAML endpoint. If authentication succeeds, the SecureSpan™ Gateway parses the redirect header and saves the "SAMLart" parameter in memory for future assertions in the same policy to use.

 

The SAML Browser Artifact assertion is useful for "mixed-mode" SAML interactions in which an initial request containing a user's credentials establishes a Single Sign-On session that can be used in subsequent browser-based requests from the same user.

 

 TIP: The saved SAML artifact value can be used in the Regular Expression assertion by entering the variable "${samlBrowserArtifact.artifact}" in the Replacement field in the Regular Expression Assertion form. For more information about the Regular Expression assertion, see Configure and Edit Message Validation/Transformation Assertions.

 

Add and Configure the SAML Browser Artifact Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The Configure SAML Browser/Artifact form appears. In the SAML Single Sign-On Endpoint URL field, enter the URL of a SAML identity provider endpoint that will accept an HTTP GET request with Basic credentials. The URL must include both the Single Sign-On (SSO) endpoint and the corresponding service endpoint. If the SSO system returns a 302 status code after processing, then the assertion succeeds and will proceed to process the service endpoint URL. If the SSO system returns a non-302 status code, if the target redirect URL cannot be parsed, or if the result does not match the SAML Artifact Query Parameter specified below, then the assertion will fail, possibly resulting in a failure of the policy.

 NOTE: The URL specified in the SAML Single Sign-On Endpoint URL field is only for authentication, not message routing. Even if the URL is the same as the endpoint of the service, the HTTP(S) Routing assertion is still required to route service messages. For more information about message routing, see Configure and Edit Message Routing Assertions.

  1. The SAML Artifact Query Parameter field is populated with the default "SAMLart" value. SAMLart is the type of cookie returned from the SSO system, which is then sent on to the target URL. If the return cookie is not the same type as the value in the SAML Artifact Query Parameter, then the assertion will fail.

 NOTE: The SAMLart parameter should suffice for most usage scenarios of the SAML Browser Artifact assertion. The default value should only be changed if the administrator of the Single Sign-On system has chosen a different parameter name. The value in the SAML Artifact Query Parameter field is case sensitive.

  1. Click [OK]. The assertion is added to the Policy Development window.

Edit the SAML Browser Artifact Assertion

  • In the Policy Development window, right-click the "Retrieve SAML Browser Artifact from..." assertion. A drop-down menu appears. Select View/Edit SAML Browser Artifact Properties

  • The Configure SAML Browser/Artifact form appears. Edit the assertion as required. See Add and Configure the SAML Browser Artifact Assertion for instructions.
