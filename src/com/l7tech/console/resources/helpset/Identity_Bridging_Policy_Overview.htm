Configure SAML Security Policies

 NOTE: The generic "Trusted Gateway A" and "Federated Gateway B" references are used in the identity bridging examples, workflows, and instructions. The Trusted Gateway A is the certificate authority (CA) that issues and manages security credentials and that is responsible for authentication. The Federated Gateway B is the Web service provider that is responsible for authorization.

As part of the required SAML credential source workflow, a SecureSpan™ Manager security policy must be configured for the shared Web service with:

  • The SAML Security assertion from the XML Security category of the [Assertions] tab configured with the holder-of-key (HOK) confirmation method

  • The federated identities (users, groups, and/or virtual groups) containing credentials shared by the Trusted Gateway A and Federated Gateway B.

 

 NOTE: See Identity Bridging Workflow Using SAML for more information about the SAML workflow.

 

In the Manager, you can only configure a security policy for a published Web service. See Manage SOAP Web Services for more information. After adding the SAML Security assertion and federated users, groups, and/or virtual groups to the security policy, configure additional security policy assertions for the Web service as outlined in Security Policy Assertions Overview and Configure and Edit Security Policies.

Configure a SAML Security Policy

The SecureSpan™ Manager Policy Development window is the repository for the assertions used to develop a security policy. Most assertions require configuration either before or after being added to the Policy Development window. When all or most of the required assertions have been added and configured in the Policy Development window, then they must be organized into the logical structure defined below and/or in Security Policy Organization and Rules. For more information about the security policy configuration process, or for information about the other security policy assertions, see Configure and Edit Security Policies and Security Policy Assertions Overview, respectively.

 

 TIP: Before constructing a security policy, ensure that the Policy Validation Messages window is enabled. To enable the window, select [View] > Policy Messages from the Main Menu (check box checked). The Policy Validation Messages window provides you with policy and assertion-level confirmation, warning, or error messages that you can use during the configuration process. See the "Validate a Security Policy" section in Configure and Edit Security Policies for more information.

 

  1. Access the Policy Development window for the target Web service by one of two methods:

 NOTE: Only published Web services appear in the [Services] tab. See Manage SOAP Web Services for more information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

  1. The Policy Development window appears. Add the SAML Security assertion to the security policy by one of two methods:

 NOTE: For more information about the SAML Security assertion, see Configure and Edit Authentication Assertions.

    • Drag and drop the SAML Security assertion from the Authentication category of the [Assertions] tab into the Policy Development window, or  

    • Highlight the SAML Security assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. Add a Grant Access to Users/Groups assertion to the security policy by one of two methods:

    • Drag and drop the Grant Access to Users/Groups assertion from the Authentication category of the [Assertions] tab into the Policy Development window, or  

    • Highlight the Grant Access to Users/Groups assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The Search Identity Provider form appears. Add the shared federated user configured in Identity Bridging Workflow Using SAML to the security policy as follows:

    • Select the Federated Identity Provider name from the Search drop-down list

    • Select the "Users" search target from the Type drop-down list

    • If necessary, specify any search criteria using the Name drop-down list and field

 TIP: The Search Identity Provider form supports the following wild card characters in the Name field:

  • The asterisk wild card (*) retrieves terms with a common initial or subsequent spelling. It matches any string of zero or more characters

  • The question mark wild card (?) substitutes one character in a term. Use within a word or at the end of a word.

    • Click [Search].

 NOTE: To stop a search that is in progress, click [Stop].

  1. Search results appear in the Search Results window. Select the target federated user, such as "user A," from the Search Results window by one of two methods:

    • Double-click the target federated user name, or

    • Select the target federated user name and click [Select].

  1. Repeat to add additional federated users, groups, and/or virtual groups as required.

 NOTE: A security policy with more than one Grant Access to Users/Groups assertion must organize the identities into "At least one assertion must evaluate to true" assertion folders. To add a folder to the Policy Development window, right-click any assertion and select "Add 'One or More' Assertion" from the drop-down menu. Drag and drop federated users, groups, and/or virtual groups into the individual folders as required. For more information, see Security Policy Organization and Rules and Configure and Edit Security Policies.

  1. Configure additional security policy assertions for the Web service as outlined in Security Policy Assertions Overview and Configure and Edit Security Policies. When completed, click [Save] in the Policy Tool Bar. Note any error or warning messages in the Policy Validation Messages window. If necessary, correct policy errors as outlined in the "Validate a Security Policy" section in Configure and Edit Security Policies.

Proceed to configure the required proxies in the SecureSpan™ Bridge. See Add Gateway Proxies for more information.
