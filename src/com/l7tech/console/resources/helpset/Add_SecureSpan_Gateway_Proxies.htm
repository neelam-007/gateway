Add SecureSpan™ Gateway Proxies

 NOTE: The generic "Trusted Gateway A" and "Federated Gateway B" references are used in the identity bridging examples, workflows, and instructions. The Trusted Gateway A is the certificate authority (CA) that issues and manages security credentials and that is responsible for authentication. The Federated Gateway B is the Web service provider that is responsible for authorization.

There are two ways to configure the SecureSpan™ Bridge to participate in an identity bridging arrangement:

  • If the Bridge has access to two SecureSpan™ Gateways — one as the Trusted Gateway A in the authentication domain and the other as the Federated Gateway B in the authorization domain — then configure the Bridge using the Full Identity Bridging method below. This method for creating proxies can be used for both the SAML and X.509 certificate credential sources

  • If the Trusted Gateway A in the identity bridging configuration is not an actual SecureSpan™ Gateway, then configure the Bridge using the Ad-Hoc Identity Bridging method below. This method for creating a proxy can only be used for the X.509 certificate credential source.

For more information about the SAML and X.509 certificate credential source workflows, see Identity Bridging Workflow Using SAML and Identity Bridging Workflow Using an X.509 Certificate, respectively.

Full Identity Bridging Method

 NOTE: You can use the following method when configuring identity bridging using either the SAML or X.509 certificate credential source. The following instructions assume that the SecureSpan™ Bridge is open.

The following items are required to configure the Gateway proxies using the Full Identity Bridging method in the SecureSpan™ Bridge:

  • To configure the Trusted Gateway proxy, you must have:

    • The host name for the Trusted Gateway A

    • SAML Credential Source: The user name and password for the Internal Identity Provider (IIP) user configured in the Trusted Gateway A and mirrored as a federated user for the Federated Identity Provider in the Federated Gateway B. See Identity Bridging Workflow Using SAML for more information

    • X.509 Credential Source: If using the Full Identity Bridging method in an configuration that uses the X.509 credential source, then you must create a shared federated user between the Trusted Gateway A and Federated Gateway B. To do so, follow the instructions outlined in Identity Bridging Workflow Using SAML before you configure the Bridge. Perform Steps #2-4 to create the IIP user in the Trusted Gateway A, then perform Step #7 to create the shared federated user in the Federated Gateway B. When completed, return to this section to configure the Bridge.

  • To configure the Federated Gateway proxy, you must have:

    • The host name for the Federated Gateway B

    • The previously configured Trusted Gateway proxy entry, uniquely identified by its proxy ID, host name, and user name.

In the SecureSpan™ Bridge, Gateway proxies are added using the Gateway Properties form.

 

  1. Access the Gateway Properties form by one of two methods:

    • Select [File] > New from the Main Menu, or

    • Click the [New] button on the Main Tool Bar.

  1. The Gateway Properties form appears. In the Gateway Host Name field in the [General] tab, enter the host name and domain of the Trusted Gateway A. For example: trustedmachinename.domain.com.

  2. Click the [Identity] tab. The Trusted Gateway option is selected by default. Enter the user name of the shared federated user into the User Name field. For example, "user A."

 NOTE: The shared federated user is the Internal Identity Provider user configured in the Trusted Gateway A and mirrored as a federated user for the Federated Identity Provider in the Federated Gateway B.

  1. Enter the password for the shared federated user into the Password field.

 NOTE: The password for the shared federated user was configured in the Internal Identity Provider user record in the Trusted Gateway A.

  1. The password entered in the Password field will remain as long as the SecureSpan™ Bridge is open on your system. To configure the Bridge to save the password in the Password field after a shutdown, check the optional Save Password check box.

 NOTE: Since one or more client certificates are used in an identity bridging configuration, do not check the "Use Credentials from Client (HTTP Basic Authentication)" check box.

  1. Click [OK]. The Trusted Gateway proxy is added to the Gateway list. Note the Trusted Gateway proxy ID, such as "gateway2." Access the Gateway Properties form to configure the Federated Gateway proxy as outlined in Step #1 above.

  2. In the [General] tab of the Gateway Properties form, enter the URL of the Federated Gateway B into the Gateway Host Name field. For example: federatedmachinename.domain.com.

  3. Click the [Identity] tab. Select the Federated Gateway option.

  4. Select the proxy ID from the Trusted Gateway drop-down list.

  1. Click [OK]. The Federated Gateway proxy is added to the Gateway list. You now can use the Federated Gateway proxy to consume the shared Web service as outlined in Consume Shared Web Services.

Ad-Hoc Identity Bridging Method

 NOTE: You can only use the following method when configuring identity bridging with the X.509 certificate credential source. The following instructions assume that the SecureSpan™ Bridge is open.

The following items are required to configure the Gateway proxy using the Ad-Hoc Identity Bridging method in the SecureSpan™ Bridge:

  • The host name for the Federated Gateway B

  • Access to a pre-existing private key PKCS#12 keystore certificate (such as "cert.p12"). The certificate can be self-signed or signed by a CA trusted by the Federated Gateway B. You must also have the PKCS#12 file password.

In the SecureSpan™ Bridge, Gateway proxies are added using the Gateway Properties form.

 

  1. Access the Gateway Properties form by one of two methods:

    • Select [File] > New from the Main Menu, or

    • Click the [New] button on the Main Tool Bar.

  1. The Gateway Properties form appears. In the Gateway Host Name field in the [General] tab, enter the URL of the Federated Gateway B. For example: https://federatedmachinename.domain.com:8443.

  2. Click the [Identity] tab. Select the Trusted Gateway option.

 NOTE: Leave the User Name and Password fields blank. The Gateway Properties form will automatically populate these values after the PKCS#12 keystore certificate has been imported below.

  1. Click the [View Client Certificate] button. The Client Certificate Not Found form appears.

  2. In the Client Certificate Not Found form, click [Import]. The Select Client Certificate form appears.

  3. Navigate to the location of the certificate. Select the file, and then click [Import Certificate].

  4. The Enter Password form appears. Enter the PKCS#12 file password into the field. Click [OK].

  5. One of two messages will appear:

    • If the password is correct, an import confirmation message appears. Click [OK]. Proceed to Step #9 below

    • If the password is incorrect, an error message will appear. Click [OK], and then repeat Steps #4-7 above. When the certificate has been imported successfully, proceed to Step #9 below.

  6. The User Name and Password fields are automatically populated with the corresponding values from the imported certificate. The password entered in the Password field will remain as long as the SecureSpan™ Bridge is open on your system. To configure the Bridge to save the password in the Password field after a shutdown, check the optional Save Password check box.

 NOTE: Since one or more client certificates are used in an identity bridging configuration, do not check the "Use Credentials from Client (HTTP Basic Authentication)" check box.

  1. Click [OK]. The Trusted Gateway proxy is added to the Gateway list. You now can use the Trusted Gateway proxy to consume the shared Web service as outlined in Consume Shared Web Services.
