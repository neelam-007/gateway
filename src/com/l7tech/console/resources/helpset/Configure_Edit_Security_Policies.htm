Configure and Edit Security Policies

 TIP: The type and order of assertions are important when constructing a valid security policy. Become familiar with the policy and assertion information and rules in Security Policy Organization and Rules before constructing a security policy.

In the SecureSpan™ Manager, a security policy is constructed for a published service by moving assertions from the [Assertions] tab into the Policy Development window. The policy assertions in the Policy Development window are organized into a logical tree-structure that conforms to the policy and assertions rules discussed in Security Policy Organization and Rules. A security policy is configured in four general steps:

 

 NOTE: Web services and XML applications are collectively referred to as "services." In the [Services] tab, Web services are identified by an  icon and XML applications are identified by an  icon. You can only configure a security policy for a published service. For information on publishing services, see Services Overview.

  1. Select the service.

  2. Add and configure the assertions.

  3. Organize the assertions.

  4. Validate the security policy.

A logical security policy must be constructed for each published service in the Manager. Policies can also be imported from or exported to a local file. See Import or Export Security Policies for more information.

 

 NOTE: For in-depth information about the category and individual assertions used in a security policy, see Security Policy Assertions Overview. For information about the interface areas related to services and security policies, see SecureSpan™ Manager Interfaces.

Configure a New Security Policy

 TIP: Reference the security policy examples in Security Policy Examples when constructing your own policies.

The SecureSpan™ Manager Policy Development window is the repository for the assertions used to develop a security policy. Most assertions require configuration either before or after being added to the Policy Development window. See Security Policy Assertions Overview for individual assertion information and instructions by assertion category. When all or most of the required assertions have been added and configured in the Policy Development window, then they must be organized into the logical structure defined in the individual assertion instructions and/or in Security Policy Organization and Rules. Once organized, proceed to Validate a Security Policy to finalize the security policy.

 

 TIP: Before constructing a security policy, ensure that the Policy Validation Messages window is enabled. To enable the window, select [View] > Policy Messages from the Main Menu (check box checked). The Policy Validation Messages window provides you with policy and assertion-level confirmation, warning, or error messages that you can use during the configuration process. See Validate a Security Policy for more information.

 

  1. Access the Policy Development window for the target service by one of two methods:

 TIP: If not already disabled, it is strongly recommended that you disable a service while constructing its security policy to prevent unsecured and unauthorized access. A disabled service contains a red "X" through its left-side icon. See Manage Published Services for information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

  1. The Policy Development window appears. Add assertions to the security policy by one of two methods:

 NOTE: When you publish a service using the Publish Web Service Wizard, Create WSDL Wizard, or Publish XML Application Wizard, the SecureSpan™ Manager automatically adds the service URL specified during the publication process as an HTTP(S) Routing assertion in the service's Policy Development window. For information about the HTTP(S) Routing assertion, see Configure and Edit Message Routing Assertions.

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

 NOTE: A configuration form will appear for some assertions before they are added to the Policy Development window. Other assertions will have to be configured after they are added, and some assertions will not require configuration. Before adding a particular assertion to the Policy Development window, see the Security Policy Assertions Overview for the category page that contains information about the assertion.

  1. Repeat to add and configure additional assertions. When completed, organize the assertions using the following tools:

 TIP: Utilize the feedback messages in the Policy Validation Messages window when constructing a security policy. See Validate a Security Policy for more information. To view a read-only presentation of the assertions by each configured identity (Grant Access to Users/Groups assertion), click the [Identity View] button on the Policy Tool Bar. To switch back to the Policy Development window, click the [Policy View] button.

Move an Assertion Up

    • To move an assertion up in the policy, either:

      • Select the assertion. Click the  (Move Up) button in the Assertion Tool Bar, or

      • Right-click the target assertion. A drop-down menu appears. Select Move Assertion Up.

    • The assertion moves up one line in the policy.

Move an Assertion Down

    • To move an assertion down in the policy, either:

      • Select the assertion. Click the  (Move Down) button in the Assertion Tool Bar, or

      • Right-click the target assertion. A drop-down menu appears. Select Move Assertion Down.

    • The assertion moves down one line in the policy.

Delete an Assertion

    • To delete an assertion from the policy, either:

      • Select the assertion. Click the  (Delete) button in the Assertion Tool Bar, or

      • Right-click the target assertion. A drop-down menu appears. Select Delete Assertion.

    • A pop-up confirmation message appears. Click [Yes].

Add an "All" Folder

To organize one or more assertion(s) into an "All assertions must evaluate to true" assertion folder:

    • Right-click any assertion. A drop-down menu appears. Select Add 'All' Assertion

    • An "All assertions must evaluate to true" folder is added to the Policy Development window. To apply the requirement to an assertion, drag and drop the assertion from the Policy Development window into the folder. Repeat to add additional assertions.

 NOTE: For more information about using assertion folders, see Security Policy Organization and Rules. To remove an assertion from a folder, drag and drop the assertion from the folder back into the Policy Development window.

Add a "One or More" Folder

To organize one or more assertion(s) into an "At least one assertion must evaluate to true" assertion folder:

    • Right-click any assertion. A drop-down menu appears. Select Add 'One or More' Assertion

    • An "At least one assertion must evaluate to true" folder is added to the Policy Development window. To apply the requirement to an assertion, drag and drop the assertion from the Policy Development window into the folder. Repeat to add additional assertions.

 NOTE: For more information about using assertion folders, see Security Policy Organization and Rules. To remove an assertion from a folder, drag and drop the assertion from the folder back into the Policy Development window.

  1. Do the following:

    • Proceed to Final Policy Validation to perform an optional but recommended final validation check on the policy. When the security policy passes the validation process, enable the service, if applicable, as outlined in Manage Published Services

    • Optional: Export the policy. See Import or Export Security Policies for information.

Validate a Security Policy

 NOTE: Validation is an optional but recommended step in the security policy configuration or revision process.

The SecureSpan™ Manager provides users with two sources of security policy validation:

  • Automatic feedback messages in the Policy Validation Messages window when [View] > Policy Messages from the Main Menu is enabled (check box checked)

 NOTE: In the SecureSpan™ Manager, the Policy Validation Messages window is enabled by default.

  • Final policy validation using the [Save] or [Validate] button on the Policy Tool Bar.

Configuration Process Feedback

The Policy Validation Messages window provides users with valuable feedback messages during the security policy configuration process. The window displays three types of messages:

Confirmation Messages

Confirmation messages confirm the addition and proper configuration of an assertion.

Warning Messages

Warning messages warn that an assertion:

  • Has been duplicated

  • Requires the presence of another assertion to be valid

  • Has been improperly configured.

Error Messages

Error messages warn that an assertion:

  • Has not been configured

  • Has been improperly located in the policy

  • Is in conflict with another assertion.

Use the Policy Validation Messages window as a source of guidance during the security policy configuration process.  

Final Policy Validation

Click the [Save] or [Validate] button on the Policy Tool Bar, or click [File] > Validate from the Main Menu, to perform an optional but recommended final validation check on a new or modified security policy. Validation yields one of two types of messages in the Policy Validation Messages window:

 

  1. A "Policy validated ok" message indicates that the policy is error-free. Proceed to enable and optionally export the policy as outlined in Manage Published Services and Import or Export Security Policies, respectively.

  2. One or more error messages indicates that the policy contains errors. Do the following:

    • For assertion organization errors, re-organize the policy assertions so that they conform to the policy and assertion organization, dependency, and combination rules outlined in Security Policy Organization and Rules and from Security Policy Assertions Overview

    • For assertion configuration errors, ensure that the individual assertions are configured properly. See the assertion-specific instructions outlined in the category pages discussed in Security Policy Assertions Overview

    • If the error message "The corresponding identity cannot be found. Please remove the assertion from the policy." appears, follow the instructions in the Invalid Users or Groups below

    • If the warning message "The assertion might not work as configured. There is no protected service JMS queue defined." appears, follow the instructions in the Invalid JMS Queue below.

 

After correcting the errors, click the [Validate] button on the Policy Tool Bar or click [File] > Validate from the Main Menu to re-validate the security policy. If a "Policy validated ok" message appears, proceed to enable and optionally export the policy as outlined in Manage Published Services and Import or Export Security Policies, respectively. If validation errors still occur, contact a Technical Support Representative as outlined in Technical Support and Contact Information.

Invalid Users or Groups

If validating an existing policy, an error message will appear in the Policy Validation Messages window if a user or group included in the security policy has been deleted from the External Identity Provider (EIP), Federated Identity Provider (FIP), or Internal Identity Provider (IIP). Do one of the following:

  • If you no longer want the identity to remain in the policy, then delete the user or group as outlined Edit a Security Policy

  • If you want the user or group to remain in the policy, then you must:

    1. Remove the user or group as outlined in Edit a Security Policy.

    2. Re-add the user or group to the appropriate IIP or FIP as outlined in Add and Edit Internal Identity Provider Users and Groups or Add and Edit Federated Identity Provider Users and Groups, respectively. Since EIP users and groups are defined outside of the SecureSpan™ Manager, use the appropriate external management program to re-add the missing EIP user or group.

    3. Use the Grant Access to Users/Groups assertion from the Authentication category of the [Assertions] tab to re-add the user or group into the target policy as outlined in Configure and Edit Authentication Assertions.

    4. Click the [Validate] button or [Save] button on the Policy Tool Bar to perform the final validation check.

When constructing a new policy, you will not be able to add a user or group that is not in the target EIP, FIP, or IIP.

Invalid JMS Queue

 NOTE: JMS Routing is an optional add-on assertion that is disabled in standard versions of the SecureSpan™ Solution. See Optional Add-On Assertions Overview for more information.

When JMS routing is used in a new or existing security policy, the validation process checks the outbound queue attached to the JMS Routing assertion. An error message will appear in the Policy Validation Messages window if the queue is unspecified or invalid. See Using the JMS Message Routing Assertion to:

  • Enter or re-enter the outbound JMS queue

  • Test the outbound JMS queue

  • Edit the JMS Routing assertion, if necessary, for the new or revised queue.

Edit a Security Policy

 NOTE: Ensure that policy changes conform to the policy and assertion rules outlined in Security Policy Organization and Rules. When editing a security policy, use the confirmation, warning, and/or error messages in the Policy Validation Messages window as a guide. To enable the Policy Validation Messages window, select [View] > Policy Messages from the Main Menu (check box checked). See Validate a Security Policy for more information.

 

  1. Access the Policy Development window for the target service by one of two methods:

 TIP: It is strongly recommended that you disable a service while editing its security policy to prevent unsecured and unauthorized access. A disabled service contains a red "X" through its left-side icon. See Manage Published Services for information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

  1. The Policy Development window appears:

    • If necessary, re-organize the assertions as outlined in Step #3 in Configure a Security Policy

    • If necessary, modify an editable assertion. See Security Policy Assertions Overview for individual assertion information and instructions by assertion category.

 NOTE: The changes that you make to an individual assertion may affect the validity of the security policy. Reference the policy and assertion rules in Security Policy Organization and Rules and from Security Policy Assertions Overview before you edit the assertion.

  1. Do the following:

    • Proceed to Final Policy Validation to perform an optional but recommended final validation check on the policy. When the security policy passes the validation process, enable the service, if applicable, as outlined in Manage Published Services

    1. Optional: Export the policy. See Import or Export Security Policies for information.
