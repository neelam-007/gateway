Comment

The Comment assertion allows you to insert a comment at any point in a policy path or within a policy folder in the Policy Development window. The Comment assertion is useful for self-documenting complex polices, especially those with nested assertions and policy folders. The Comment assertion has no effect on the runtime processing of a policy.

Add and Configure the Comment Assertion

  1. In the Policy Development window, select an assertion or policy folder corresponding to the target insert location for the Comment assertion. The comment will appear after the selected item.

  1. Insert the assertion by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The Comment Assertion Properties form appears. Type the comment text into the Comment field.

NOTE: The text does not require comment characters "<!--comment text-->."

  1. Click [OK]. The assertion is added to the Policy Development window.

Edit the Comment Assertion

  • In the Policy Development window, right-click the "Comment:" assertion. A drop-down menu appears. Select Comment Assertion Properties

  • The Comment Assertion Properties form appears. Edit the assertion properties as outlined in Add and Configure the Comment Assertion.

 

Equality

The Equality assertion compares two values during the runtime processing of a policy. Both comparison values can be context variables in the format "${variable.name}" (where "variable.name" is the context variable), or the comparison values can be made up of one context variable and one constant. A policy can contain more than one Equality assertion.

 

For example, when used with two XPath assertions — the Evaluate Request XPath assertion and/or the Evaluate Response XPath assertion (see Configure and Edit Message Validation/Transformation Assertions) — the Equality assertion could be used to ensure that the value of a particular attribute in a SAML attribute statement (set with the SAML Assertion discussed in Configure and Edit Access Control Assertions) matches a value found in the message body. In this scenario, at least one of the XPath assertions must have a non-default variable prefix so that the second XPath assertion does not overwrite the variables set by the first. Then, the Equality assertion must be configured to compare the two ".result" variables and placed after the SAML Assertion and two XPath assertions in the Policy Development window. In another example scenario, the Equality assertion could be used to verify that exactly one node has been found by a previous Evaluate Request XPath assertion by testing that ${requestXpath.count} is equal to "1".

Add and Configure the Equality Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The Equality form appears. Enter the first value into the Expression 1 field. Within the expression, variables can be referenced using ${assertionPrefix.variableName} syntax.

NOTE: You can mix variable and constant values in the Expression 1 and Expression 2 fields by surrounding variable references with additional text. For example, if the variable "requestXpath.result" contains the value "B" and the expression entered is A${requestXpath.result}C, then the resulting value that will be compared by the Equality assertion is "ABC".

  1. Enter the second value into the Expression 2 field. Click [OK].

  2. The assertion is added to the Policy Development window.

Edit the Equality Assertion

  • In the Policy Development window, right-click the "Proceed if:" assertion. A drop-down menu appears. Select Equality Assertion Properties

  • The Equality form appears. Edit the assertion properties as outlined in Add and Configure the Equality Assertion.

 

At Least One Assertion Must Evaluate to True

TIP: If you do not want the failure of an "At least one assertion must evaluate to true" folder to fail the entire policy, then add a Continue Processing assertion into the assertion folder (see Continue Processing). The Continuing Processing assertion will always evaluate to true, preventing the failure of a policy due to the failure of a non-essential or conditional assertion in an "At least one assertion must evaluate to true" folder.

The "At least one assertion must evaluate to true" assertion folder organizes and defines the processing conditions for the assertions that it contains and for the overall policy. When assertions are grouped into an "At least one assertion must evaluate to true" folder in the Policy Development window, each successive child assertion is processed until a single assertion succeeds, yielding a success outcome for the folder. If all child assertions in an "At least one assertion must evaluate to true" folder fail, then the overall folder—and possibly the entire policy—fails. For more information about parent and child assertions in a policy, see Policy Organization.

Add and Configure the At Least One Assertion Must Evaluate to True Folder

  1. Add an "At least one assertion must evaluate to true" assertion folder to the Policy Development window by one of two methods:

Use the Right-Click Menu

Add the assertion folder from the Policy Development window's right-click menu:

    • Highlight an assertion near the desired insert location for the assertion folder

    • Right-click the Policy Development window. A drop-down menu appears

    • Select Add 'One or More' Assertion.

Insert from the [Assertions] Tab

Add the assertion folder from the [Assertions] tab by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The assertion folder is added to the Policy Development window. Drag and drop assertions from the Policy Development window into the assertion folder, or add new assertions directly into the assertion folder as required:

TIP: To add a new identity into the assertion folder, right-click the assertion folder in the Policy Development window. A drop-down menu appears. Select Add User or Group. Proceed to Configure and Edit Access Control Assertions to configure the new Grant Access to Users/Groups assertion. Note that there should not be more than one first-level Grant Access to Users/Groups assertion in a policy or within an assertion folder. To grant access to more than one user or group in a policy, place each individual Grant Access to Users/Groups assertion in an individual "At least one assertion must evaluate to true" or "All assertions must evaluate to true" folder with an authentication assertion (and other assertions, as required). See Policy Organization for more information.

    • For assertion configuration instructions, view the category specific instruction pages outlined in About Policy Assertions

    • For assertion folder information and examples, see Policy Organization

    • For assertion and assertion folder organization instructions, see Configure and Edit Policies.

  1. Repeat to add and organize additional assertion folders.

Edit an At Least One Assertion Must Evaluate to True Folder

The contents and location of an "At least one assertion must evaluate to true" assertion folder can be modified at any time:

  • New assertions can be added to the folder by dragging and dropping them from the Policy Development window or adding from the [Assertions] tab

  • Assertions can be removed from the folder by dragging and dropping them back into the Policy Development window

  • The location of the assertion folder in a policy, and the order of the assertions within the folder, can be modified as outlined in Configure and Edit Policies.

Delete an At Least One Assertion Must Evaluate to True Folder

WARNING: Deleting an assertion folder also removes the assertions that it contains from the Policy Development window and the SecureSpan™ Gateway, possibly invalidating the policy.

To delete an "At least one assertion must evaluate to true" assertion folder:

  1. Right-click the folder in the Policy Development window. A drop-down menu appears.

  2. Select Delete Assertion. A confirmation message appears.

  3. Click [Yes]. The assertion folder and its contents are removed from the Policy Development window and the SecureSpan™ Gateway.

 

All Assertions Must Evaluate to True

The "All assertions must evaluate to true" assertion folder organizes and defines the processing conditions for the assertions that it contains and for the overall policy. When assertions are grouped into an "All assertions must evaluate to true" folder, each successive child assertion is processed until all assertions succeed, yielding a success outcome for the folder. Processing in the "All assertions must evaluate to true" folder will stop when the first child assertion fails, yielding a fail outcome for the folder—and possibly the entire policy. For more information about parent and child assertions in a policy, see Policy Organization.

Continue Processing

NOTE: The Continue Processing assertion must be used with an "At least one assertion must evaluate to true" assertion folder.

The Continue Processing assertion is a placeholder assertion that will always yield a successful or true processing result. When used in an "At least one assertion must evaluate to true" assertion folder (see At Least One Assertion Must Evaluate to True), the Continue Processing assertion will ensure that the incidental failure of a non-essential assertion within the same folder will not cause an overall policy failure.

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window. For information on adding and organizing the Continue Processing assertion into an "At least one assertion must evaluate to true" assertion folder, see At Least One Assertion Must Evaluate to True.
