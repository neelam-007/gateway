Import or Export Policies

The SecureSpan™ Manager allows you to:

  • Import a policy into the Policy Development window. Use this feature to ensure policy consistency, save configuration time, and implement shared policies amongst external departments, partners, and others with separate SecureSpan™ Gateways or entire SecureSpan™ integrations. See Import a Policy for more information

  • Export a policy as an XML file. Exported policies can be saved to a SecureSpan Manager folder for local re-use or an external file location. Use this feature to share policies internally or externally or to save policy copies for record-keeping purposes. See Export a Policy for more information. Exported policies saved locally can also be renamed, deleted, or edited by replacement. See Rename, Delete, or Edit a Policy Template for more information.

 

The policies are imported into and exported from the SecureSpan Manager as portable XML files encapsulating all of the originating policy information.

Import a Policy

 WARNING: When you import a policy, the target policy in the Policy Development window is completely replaced by the imported assertions, identity provider properties (see Manage Identity Providers), JMS queue references, and any other optional add-on assertions (see Optional Add-On Assertions), if present. The imported policy must then be re-configured to reflect the unique service URL and any other items required by the imported policy. If the originating policy contains a Validate XML Schema assertion (see Configure and Edit Message Validation/Transformation Assertions) with a schema that includes one or more import statements, then you will need to resolve the external schema(s) (called global schemas) with the Manage Global Schemas table (see Manage Global Schemas). Ensure that you have service-specific information on hand for re-configuring the imported policy.

The SecureSpan™ Manager allows you to import a policy into the Policy Development window from one of two sources:

  1. If the policy was previously exported from within the SecureSpan Manager, then the policy will appear as a template in the Policy Templates section of the [Assertions] tab. Drag and drop the policy into the Policy Development window for the target service.

  2. A target policy that did not originate from a previously exported SecureSpan Manager policy can be imported from an external location into the Policy Development window by clicking the [Import Policy] button on the Policy Tool Bar.

 

The policy XML file will overwrite any assertions present in the original policy, including the target service's unique routing information. Importation also imports the referenced identity provider users and groups, JMS routing endpoints, and other add-on assertion references into the new policy.

 

  1. If not already present, load the target policy into the Policy Development window by one of the following methods:

 TIP: Disable a service while constructing its policy to prevent unsecured and unauthorized access. A disabled service contains a red "X" through its left-side icon. See Manage Published Services for information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

  1. The Policy Development window appears. If not present, enable the Policy Validation Messages window by selecting [View] > Policy Messages from the Main Menu (check box checked).

  2. Import a policy by one of the following methods:

Import the Policy from the Template Folder

 NOTE: The template folder is located in the local ".l7tech\policy.templates" policy directory. The folder stores the templates that appear in the Policy Templates folder of the [Assertions] tab.

    1. Move the target policy template from the Policy Templates section of the [Assertions] tab into the Policy Development window by one of the following methods:

      • Drag and drop the selected template from the Policy Templates section into the Policy Development window, or  

      • Select the template in the Policy Templates section and click the  (Add Assertion) button in the Assertion Tool Bar.

    1. Proceed to Step #4 below.

Import the Policy from Any File

    1. Click the [Import Policy] button on the Policy Tool Bar. The Import Policy form appears.

    2. The default ".l7tech\policy.templates" policy folder appears. Navigate to the location of the policy XML file. Select the file, and then click [Open].

 NOTE: The Import Policy form defaults to the ".l7tech\policy.templates" directory that stores policies saved as templates in the Policy Templates section of the [Assertions] tab. The SecureSpan™ Manager can only import version 3.0 and later XML policy files.

    1. Proceed to Step #4 below.

 

  1. Before importing the policy into the Policy Development window, the SecureSpan Manager automatically resolves the back-end requirements of the imported policy against the back-end configuration of the target policy's SecureSpan™ Gateway. One of the following will occur:

Automatic Reconciliation Fails

Automatic reconciliation will fail if the imported policy contains references to elements that are not present on the target system. This will typically happen if the policy came from another system or if the policy refers to elements, such as a user or group, that were deleted since the policy was originally exported. If automatic reconciliation of the back-end requirements between the imported and target policy fails, then the Resolve external dependencies from exported policy document wizard will appear. The Wizard allows you to manually reconcile the elements in the policy that failed the automatic reconciliation process:

    • If the identity provider for the users or groups attached to the imported policy does not exactly match an identity provider on the target SecureSpan Gateway, then a message appears prompting you to:

 WARNING: The SecureSpan™ Manager compares the object-level property values of the imported identity provider with each identity provider configured in the target SecureSpan™ Gateway. A difference in even one value will cause a reconciliation failure.

      • Select a local identity provider to substitute for the imported identity provider, or

      • Configure a new identity provider or edit an existing identity provider (see Manage Identity Providers) so that its property values matches the values of the imported identity provider, or

 TIP: When creating or editing an identity provider, consult the properties values of the imported identity provider found in the imported policy XML file. Navigate to the source location of the file, open the file in a text editor, and note the values in the "<exp:References>" parameter. Policies located in the Policy Templates section of the [Assertions] tab are located in a local ".l7tech\policy.templates" directory.

      • Abort the import process.

    • If JMS routing is referenced and/or if the endpoints corresponding to the configured queues in the originating policy cannot be resolved on the target SecureSpan Gateway, then a message will appear prompting you to:

      • Enable JMS routing in the target SecureSpan Gateway, if applicable, or

      • Configure the inbound and/or outbound queues required by the imported JMS routing assertion (see Using the JMS Message Routing Assertion), or

 TIP: When creating or editing a JMS queue, consult the properties values of the imported queues found in the imported policy XML file. Navigate to the source location of the file, open the file in a text editor, and note the values in the "<exp:References>" parameter. Policies located in the Policy Templates section of the [Assertions] tab are located in a local ".l7tech\policy.templates" directory.

      • Select or modify one or more existing inbound or outbound queues to match the configuration of the imported queue references, or

      • Abort the import process.

    • If one or more optional add-on assertions are referenced in the imported policy when the required back-up systems are not configured on the target SecureSpan Gateway, then a message will appear prompting you to:

      • Ignore and remove the add-on assertions from the policy (see Optional Add-On Assertions), or

      • Abort the import process.

When the back-end requirements of the imported policy have been met, return and repeat the import instructions outlined in Import a Policy. When the SecureSpan Manager successful reconciles the identity provider(s), JMS routing endpoints, and/or add-on assertion requirements from the imported policy, then the imported policy will appear in the Policy Development window. Proceed to Step #5.

Automatic Reconciliation Succeeds

If the SecureSpan Manager successful reconciles the identity provider(s), JMS routing endpoints, and/or add-on assertion requirements from the imported policy, then the imported policy will appear in the Policy Development window. Proceed to Step #5.

 

  1. The routing assertion(s) and other assertions in the Policy Development window are specific to the service that originated the policy. Edit the assertions for the target service as required:

    • See Configure and Edit Message Routing Assertions to re-configure the replaced routing assertion with service-specific information

    • See About Policy Assertions to re-configure other assertions as required.

 TIP: Note the policy configuration and organization instructions in Configure and Edit Policies when modifying the policy. Ensure that policy changes conform to the policy and assertion rules outlined in Policy Organization.

  1. Do the following:

    • Proceed to the "Validate a Policy" section in Configure and Edit Policies to perform a final validation check on the policy. When the policy passes the validation process, enable the service, if applicable, as outlined in Manage Published Services

    • If the imported policy contains a Validate XML Schema assertion (see Configure and Edit Message Validation/Transformation Assertions) that includes an import statement, then you will need to resolve the external reference(s) with the Manage Global Schemas table (see Manage Global Schemas)

    • Optional: Export the validated policy into an external file location, as a new template in the Policy Templates section of the [Assertions] tab, or use it to overwrite an existing template. See Export a Policy or Rename, Delete, or Edit a Policy Template, respectively, for more information.

 

Export a Policy

 NOTE: It is recommended that you only export valid policies. Validation confirms the proper configuration and organization of a policy. See the "Validate a Policy" section in Configure and Edit Policies for more information.

The SecureSpan™ Manager allows you to export a policy in the Policy Development window to one or more locations:

  1. A policy exported into the default local policy directory will appear as a template in the Policy Templates section of the [Assertions] tab.

  2. A policy that is exported to any other location (not to the default policy directory) will not appear in the Policy Templates section of the [Assertions] tab.

 

The portable policy XML file generated by the SecureSpan Manager includes references to:

  • The identity providers belonging to the users and groups in the policy

  • The JMS routing endpoints, or queues, if included in the policy, and

  • The add-on assertions, if included in the policy.

 

  1. If not already present, load the target policy into the Policy Development window by one of the following methods:

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

  1. The Policy Development window appears. If not present, enable the Policy Validation Messages window by selecting [View] > Policy Messages from the Main Menu (check box checked).

  2. Click the [Export Policy] button on the Policy Tool Bar. The Export Policy form appears. Do one of the following:

Export the Policy as a Template

To export the policy into the default ".l7tech\policy.templates" policy directory that stores templates that appear in the Policy Templates section of the [Assertions] tab:

 

    1. Enter a descriptive name for the policy into the File Name field.

    2. Click [Save]. The policy appears in the Policy Templates section.

To edit, rename, or delete the policy template, see Rename, Delete, or Edit a Policy Template.

Export the Policy to Any File

To export the policy to a location that is different from the default policy directory:

 NOTE: Policies that are not saved in the default policy directory will not appear in the Policy Templates section of the [Assertions] tab.

 

    1. Navigate to the target location.

    2. Enter a descriptive name for the policy into the File Name field.

    3. Click [Save]. The policy is exported as a portable XML file that encapsules all of the assertions and back-end settings.

 

 TIP: You can open the exported XML file in a text editor to view policy details. Do not change the XML when viewing the file. Doing so may invalidate the policy.

 

Rename, Delete, or Edit a Policy Template

The SecureSpan™ Manager allows you to easily rename and delete an exported policy that appears in the Policy Templates section of the [Assertions] tab. Although you cannot directly edit the contents of the XML file for an exported policy, you can "edit" the assertions in the policy template by replacement. In other words, you can overwrite the exported XML file with a new policy.

 

Rename a Policy Template

  1. In the Policy Templates section of the [Assertions] tab, right-click the target template name. A drop-down menu appears. Select Rename.

  2. The Rename Policy Template form appears. Enter the new template name into the New Template Name field.

  3. Click [OK]. The new template name appears in the Policy Templates section.

 

Delete an Exported Policy

  1. In the Policy Templates section of the [Assertions] tab, right-click the target template name. A drop-down menu appears. Select Delete.

  2. A confirmation message appears. Click [Yes]. The template is immediately removed from the Policy Templates section.

 

Edit the Assertions in an Exported Policy

  1. Following the instructions in Export a Policy to load and validate the replacement policy in the Policy Development window.

  2. Save the policy template as outlined in Export the Policy as a Template. In the Export Policy form:

    • Select the name of the policy in the ".l7tech\policy.templates" policy directory that you want to replace with the new policy

    • Click [Save]. A confirmation message appears. Click [OK].

  3. The modified template appears in the Policy Templates section.
