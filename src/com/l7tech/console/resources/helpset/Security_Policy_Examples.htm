Security Policy Examples

 TIP: The type and order of assertions are important when constructing a valid security policy. Become familiar with the policy and assertion information and rules in Security Policy Organization and Rules before constructing a security policy. For in-depth information about the category and individual assertions used in a security policy, see Security Policy Assertions Overview. For information on configuring a security policy, see Configure and Edit Security Policies.

The following security policy scenarios illustrate the policy and assertion organization, dependency, and combination rules described in Security Policy Organization and Rules. Use the scenarios as reference-only examples when learning how to construct your own security policy.

 

 NOTE: The following examples are for reference purposes only. An "identity" is a user or group configured with the Grant Access to Users/Groups authentication assertion. See Configure and Edit Authentication Assertions for more information. By default, all security policies contain a message routing assertion. See Configure and Edit Message Routing Assertions for more information. Web services and XML applications are collectively referred to as "services." See Service Overview for more information.

Scenario One: One Authentication Method/One or More Identities

In this security policy, a single authentication method is used, and only the listed users and groups have authorized access to the service. Access requests are routed to the defined HTTP(S) service URL.

Scenario Two: Multiple Authentication Methods/One or More Identities

In this security policy, multiple authentication methods are used to gain service access when the authorized identities evaluate to "true." Access requests are routed to the defined HTTP(S) service URL.

 

A Security Policy with Multiple Authentication Methods and One or More Identities

Scenario Three: Multiple Authentication Methods/One or More Identities/Response Security

In this security policy, multiple authentication methods are also used to gain service access when the authorized identities evaluate to "true." Access requests are routed to the defined HTTP(S) service URL. Responses from the service are XML signed.
