Using the JMS Message Routing Assertion

NOTE: The JMS Routing assertion and queues support the JMS 1.0 standard. The Administrator is responsible for installing and configuring the items required for JMS routing. If you encounter errors during the execution of a JMS policy, contact your Administrator.

The JMS routing assertion — called "JMS Routing" in the [Assertions] tab and "Route to JMS Queue" in the Policy Development window — allows you to configure the JMS transportation of inbound and/or outbound service messages. Before JMS routing can be used in a policy, the queues that were configured in the IBM® WebSphere® MQ (version 5.3) Queue Manager and referenced in the JNDI directory must be registered in the SecureSpan™ Manager. See About JMS Queues for more information.

About JMS Queues

One or more JMS queues must be configured in the SecureSpan™ Manager before the SecureSpan™ Gateway can use JMS to communicate with service requestors and published services. The type of queue required depends on whether the queue will be used for receiving messages from requestors (inbound) or for sending messages to a Web service or XML application (outbound).  

 

The SecureSpan Gateway automatically monitors all inbound queues configured in the SecureSpan Manager. When the SecureSpan Gateway receives a message from an inbound queue, it determines which service the message is destined for and executes the applicable policy.

 

Security policies normally include either an HTTP(S) Routing or JMS Routing assertion that specifies message forwarding using either HTTP(S) or JMS, respectively. In the SecureSpan Gateway, there are three possible HTTP(S) and JMS routing scenarios:

  • JMS-to-JMS Configuration

In a JMS-to-JMS configuration scenario, the SecureSpan Gateway receives messages from a monitored inbound JMS queue (iJMS), and then uses a JMS Routing assertion to transmit the messages to an outbound JMS queue (oJMS):

Requestor iJMS SecureSpan Gateway oJMS Service

If a response message is required by the requestor, then the SecureSpan Gateway creates a temporary queue to receive the reply from the service. The SecureSpan Gateway then routes the reply message back to the requestor in accordance with the requestor-designated reply-to queue in the header of the JMS message. The reply-to queue specified by the requestor may be temporary. If reply instructions are not included in the header of the JMS message, however, then the SecureSpan Gateway cannot route the reply back to the requestor.

  • JMS-to-HTTP Configuration

In the JMS-to-HTTP configuration scenario, the SecureSpan Gateway receives messages from a monitored inbound JMS queue (iJMS), and then transmits the messages to the service using HTTP:

Requestor iJMS SecureSpan Gateway HTTP Service

Once again, the SecureSpan Gateway monitors the inbound queue configured in the SecureSpan Manager. When the SecureSpan Gateway picks up a message from the queue, it routes the message to the service URL specified in the HTTP(S) Routing assertion in the service’s policy. The SecureSpan Gateway routes any service response message back to the requestor-designated reply-to queue specified in the header of the JMS message. If reply instructions are not included in the header of the JMS message, however, then the SecureSpan Gateway cannot route the reply back to the requestor.

NOTE: When you publish a service, the SecureSpan™ Manager automatically adds the service URL specified during the publication process as an HTTP(S) Routing assertion (see Configure and Edit Message Routing Assertions) in the policy.

  • HTTP-to-JMS Configuration

In the HTTP-to-JMS configuration scenario, the SecureSpan Gateway receives messages via HTTP, and then transmits the messages to an outbound JMS queue (oJMS):

Requestor HTTP SecureSpan Gateway oJMS Service

The SecureSpan Gateway receives a message over HTTP or HTTP(S), allowing the use of the HTTP authentication assertions (the HTTP Basic, HTTP Digest, and HTTP Client Certificate assertions discussed in Configure and Edit Access Control Assertions) in the policy. If a response message is required by the requestor, then the SecureSpan Gateway creates a temporary queue to receive the reply from the service. The SecureSpan Gateway then routes the reply message back to the requesting HTTP or HTTP(S) requestor.

 

All queues must be created in the WebSphere® MQ Queue Manager and referenced in the JNDI directory before they can be configured in the SecureSpan Manager.

 

Add and Configure a JMS Queue

NOTE: For more information on the usage of inbound and outbound JMS queues in the SecureSpan™ Gateway, see About JMS Queues. To view a list of previously configured queues, access the Manage JMS Queues form by selecting Tasks > Manage JMS Queues from the SecureSpan™ Manager Main Menu.

An inbound queue allows the SecureSpan™ Gateway to receive messages from a JMS queue, whereas an outbound queue directs the SecureSpan Gateway to route messages to a service that is listening on a JMS queue. The queues must first be created in the WebSphere® MQ Queue Manager and referenced in the JNDI directory before they can be configured in the SecureSpan™ Manager. A JMS queue can be edited or deleted at any time.

 

NOTE: For brevity, an outbound queue is used as an example in the following instructions. Repeat the instructions with the alternate selection in Step #7 to configure a new inbound queue. The configuration of a new queue requires the JNDI directory settings specified during the JMS configuration process. Consult your Administrator for the required values.

 

  1. In the SecureSpan™ Manager, select Tasks > Manage JMS Queues from the Main Menu. The Manage JMS Queues form appears.

TIP: You can also add a new queue by clicking [New Queue] in the JMS Routing Properties form. To access the form, double-click a "Route to JMS Queue" assertion in the Policy Development window.

  1. Click [Add]. The Add JMS Queue form appears. In the Queue Name field, enter the JNDI queue reference name. For example: cn=GatewayQueue. Replace “GatewayQueue” with the actual queue name.

  2. Select “WebSphere MQ over LDAP” from the Driver drop-down list.

NOTE: The SecureSpan™ Gateway will support other JMS providers in a future version.

  1. In the Naming Provider URL field, enter the URL of the JNDI server. For example: ldap://servername.company.com/dc=companydomain,dc=com.

  2. In the Queue Connection Factory URL field, enter the JNDI QueueConnectionFactory reference name. For example: cn=QueueConnectionFactoryJNDIName. Replace “QueueConnectionFactoryJNDIName” with the actual name.

  3. In the Initial Context Factory Class field, enter the initial context class. For example: com.sun.jndi.ldap.LdapCtxFactory.

  4. In the Direction section of the form, select the Outbound option.

NOTE: If configuring an inbound queue, select the Inbound option in the Direction section of the Add JMS Queue form.

  1. If WebSphere® MQ requires a user name and password, then check the Requires User Name and Password check box. Enter the user name and password into the respective User Name and Password fields.

  2. Click [Test Settings] to test the connection between the SecureSpan™ Gateway and JMS queue. A dialog form appears. Note the diagnostic information. If a connection could not be established, double-check the queue and SecureSpan Gateway settings. To contact a Layer 7 Technologies Technical Support Representative, see Technical Support. To close the form, click [X] in the top right corner of the window.

  3. Click [Save]. The Add JMS Queue form closes. The queue is registered in the SecureSpan Gateway and added to the Known JMS Queues table in the Manage JMS Queues form. For instructions on using the outbound queue in a service with JMS routing, proceed to Add and Configure the JMS Assertion.

 

WARNING: The JMS queue settings in the SecureSpan™ Manager must match the corresponding settings in the JNDI directory in order to connect to the SecureSpan™ Gateway.

 

Edit a JMS Queue

  1. In the SecureSpan™ Manager, select Tasks > Manage JMS Queues from the Main Menu. The Manage JMS Queues form appears.

  2. Click [Properties]. The JMS Queue Properties form appears. Edit the queue as required. Reference the instructions in Add and Configure a JMS Queue for field-level information.

  3. Click [Test Settings] to test the connection between the SecureSpan™ Gateway and JMS queue. A dialog form appears. Note the diagnostic information. If a connection could not be established, double-check the queue and SecureSpan Gateway settings. To contact a Layer 7 Technologies Technical Support Representative, see Technical Support. To close the form, click [X] in the top right corner of the window.

  4. Click [Save]. The JMS Queue Properties form is closed. The revised queue is updated in the SecureSpan Gateway and Known JMS Queues table in the Manage JMS Queues form.

 

Remove a JMS Queue

NOTE: When you remove a queue, you unregister it from the SecureSpan™ Gateway. The queue will still exist in the WebSphere® MQ server.

 

  1. In the SecureSpan™ Manager, select Tasks > Manage JMS Queues from the Main Menu. The Manage JMS Queues form appears.

  2. Select the queue that you want to remove from the Known JMS Queues table. Click [Remove].

  3. A confirmation message appears. Click [Remove].

 

JMS Assertion Rules

Note the following rules when using the JMS assertion for outbound routing:

  • If JMS service request messages are sent to the SecureSpan Gateway, then the following HTTP-related assertions cannot be used in the policy:

    • The IP Address Range assertion from the Service Availability category of the [Assertions] tab

    • The HTTP Basic, HTTP Digest, and SSL with Client Certificate Authentication assertions from the Access Control category of the [Assertions] tab

    • The SSL or TLS Transport assertion from the Transport Layer Security (TLS) category of the [Assertions] tab.

  • If HTTP request messages are sent to the SecureSpan Gateway, then all assertions can be used in the policy.

 

Add and Configure the JMS Assertion

NOTE: Inbound JMS queues are used by published JMS services and do not require an assertion. The JMS service publication process is the same as the publication process for a regular service (see Manage Services). If publishing a service that will accept incoming messages from a JMS queue, ensure that the appropriate inbound JMS queue is monitored for the messages. See About JMS Queues for more information. JMS messages will be processed according to the assertions defined in the policy. Consult the rules in JMS Assertion Rules before using the JMS assertion.

The JMS routing assertion — called "Route to JMS Queue" — appears in the Message Routing category of the [Assertions] tab in the SecureSpan™ Manager. The JMS assertion instructs the SecureSpan™ Gateway to route service request messages to a selected outbound queue configured in Add and Configure a JMS Queue.

  1. Access the Policy Development window for the target service by one of two methods:

TIP: Disable a service while constructing its policy to prevent unsecured and unauthorized access. A disabled service contains a red "X" through its left-side icon. See Manage Published Services for information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

  1. The Policy Development window appears. Add the JMS assertion to the policy by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or   

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The JMS Routing Properties form appears. Select the target outbound queue from the JMS Queue drop-down list.

NOTE: Click [New Queue] to create a new queue. The Add JMS Queue form appears. See Add and Configure a JMS Queue for instructions and more information.

  1. Click [OK]. The assertion is added to the Policy Development window. Proceed to the "Validate a Policy" section in Configure and Edit Policies.

 

Edit the JMS Assertion

  1. In the Policy Development window, right-click the "Route to JMS Queue" assertion. A drop-down menu appears. Select Routing Properties.

  1. The JMS Routing Properties form appears. Select a new queue from the JMS Queue drop-down list.

  1. Click [OK]. The modified assertion is added to the Policy Development window. Proceed to the "Validate a Policy" section in Configure and Edit Policies.
