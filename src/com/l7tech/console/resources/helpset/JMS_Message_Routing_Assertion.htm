Using the JMS Message Routing Assertion

 NOTE: The JMS Routing assertion and queues support the JMS 1.0 standard. The Administrator is responsible for installing and configuring the items required for JMS routing. If you encounter errors during the execution of a JMS security policy, contact your Administrator.

The JMS routing assertion — called "JMS Routing" in the [Assertions] tab and "Route to JMS Queue" in the Policy Development window — allows you to configure the JMS transportation of inbound and/or outbound service messages. Before JMS routing can be used in a security policy, the queues that were configured in the IBM WebSphere® MQ (version 5.3) Queue Manager and referenced in the JNDI directory must be registered in the Manager. See About JMS Queues for more information.

About JMS Queues

 NOTE: Web services and XML applications are collectively referred to as "services."

One or more JMS queues must be configured in the SecureSpan™ Manager before the SecureSpan™ Gateway can use JMS to communicate with service requestors and published services. The type of queue required depends on whether the queue will be used for receiving messages from requestors (inbound) or for sending messages to a Web service or XML application (outbound).  

 

The Gateway automatically monitors all inbound queues configured in the Manager. When the Gateway receives a message from an inbound queue, it determines which service the message is destined for and executes the applicable security policy.

 

Security policies normally include either an HTTP(S) Routing or JMS Routing assertion that specifies message forwarding using either HTTP(S) or JMS, respectively. In the Gateway, there are three possible HTTP(S) and JMS routing scenarios:

 

  • JMS-to-JMS Configuration

In a JMS-to-JMS configuration scenario, the Gateway receives messages from a monitored inbound JMS queue (iJMS), and then uses a JMS Routing assertion to transmit the messages to an outbound JMS queue (oJMS):

Requestor iJMS Gateway oJMS Service

If a response message is required by the requestor, then the Gateway creates a temporary queue to receive the reply from the service. The Gateway then routes the reply message back to the requestor in accordance with the requestor-designated reply-to queue in the header of the JMS message. The reply-to queue specified by the requestor may be temporary. If reply instructions are not included in the header of the JMS message, however, then the Gateway cannot route the reply back to the requestor.

  • JMS-to-HTTP Configuration

In the JMS-to-HTTP configuration scenario, the Gateway receives messages from a monitored inbound JMS queue (iJMS), and then transmits the messages to the service using HTTP:

Requestor iJMS Gateway HTTP Service

Once again, the Gateway monitors the inbound queue configured in the Manager. When the Gateway picks up a message from the queue, it routes the message to the service URL specified in the HTTP(S) Routing assertion in the service’s security policy. The Gateway routes any service response message back to the requestor-designated reply-to queue specified in the header of the JMS message. If reply instructions are not included in the header of the JMS message, however, then the Gateway cannot route the reply back to the requestor.

 

 NOTE: When you publish a service, the SecureSpan™ Manager automatically adds the service URL specified during the publication process as an HTTP(S) Routing assertion in the service’s security policy. For more information, see Configure and Edit Message Routing Assertions.

 

  • HTTP-to-JMS Configuration

In the HTTP-to-JMS configuration scenario, the Gateway receives messages via HTTP, and then transmits the messages to an outbound JMS queue (oJMS):

Requestor HTTP Gateway oJMS Service

The Gateway receives a message over HTTP or HTTP(S), allowing the use of the HTTP authentication assertions (HTTP Basic, HTTP Digest, and HTTP Client Certificate) in the security policy. If a response message is required by the requestor, then the Gateway creates a temporary queue to receive the reply from the service. The Gateway then routes the reply message back to the requesting HTTP or HTTP(S) requestor.

 

 NOTE: For more information about the HTTP authentication assertions, see Configure and Edit Authentication Assertions.

 

All queues must be created in the WebSphere® MQ Queue Manager and referenced in the JNDI directory before they can be configured in the Manager.

Configure and Edit JMS Queues

 NOTE: For more information on the usage of inbound and outbound JMS queues in the SecureSpan™ Gateway, see About JMS Queues. To view a list of previously configured queues, access the Manage JMS Queues form by selecting Tasks > Manage JMS Queues from the Main Menu.

An inbound queue allows the SecureSpan™ Gateway to receive messages from a JMS queue, whereas an outbound queue directs the Gateway to route messages to a service that is listening on a JMS queue. The queues must first be created in the WebSphere® MQ Queue Manager and referenced in the JNDI directory before they can be configured in the Manager. A JMS queue can be edited or deleted at any time.

Configure a New JMS Queue

 NOTE: The following instructions assume that the SecureSpan™ Manager is open. For brevity, an outbound queue is used as an example in the following instructions. Repeat the instructions with the alternate selection in Step #7 to configure a new inbound queue. The configuration of a new queue requires the JNDI directory settings specified during the JMS configuration process. Consult your Administrator for the required values.

 

  1. In the SecureSpan™ Manager, select Tasks > Manage JMS Queues from the Main Menu. The Manage JMS Queues form appears.

 NOTE: You can also add a new queue by clicking [New Queue] in the JMS Routing Properties form. To access the form, double-click a "Route to JMS Queue" assertion in the Policy Development window.

  1. Click [Add]. The Add JMS Queue form appears. In the Queue Name field, enter the JNDI queue reference name. For example: cn=GatewayQueue. Replace “GatewayQueue” with the actual queue name.

  2. Select “WebSphere MQ over LDAP” from the Driver drop-down list.

 NOTE: The SecureSpan™ Gateway will support other JMS providers in a future version.

  1. In the Naming Provider URL field, enter the URL of the JNDI server. For example: ldap://servername.company.com/dc=companydomain,dc=com.

  2. In the Queue Connection Factory URL field, enter the JNDI QueueConnectionFactory reference name. For example: cn=QueueConnectionFactoryJNDIName. Replace “QueueConnectionFactoryJNDIName” with the actual name.

  3. In the Initial Context Factory Class field, enter the initial context class. For example: com.sun.jndi.ldap.LdapCtxFactory.

  4. In the Direction section of the form, select the Outbound option.

  5. If WebSphere® MQ requires a user name and password, then check the Requires User Name and Password check box. Enter the user name and password into the respective User Name and Password fields.

  6. Click [Test Settings] to test the connection between the SecureSpan™ Gateway and JMS queue. A dialog form appears. Note the diagnostic information. If a connection could not be established, double-check the queue and Gateway settings. To contact a Layer 7 Technologies Technical Support Representative, see Technical Support and Contact Information. To close the form, click [X] in the top right corner of the window.

  7. Click [Save]. The Add JMS Queue form closes. The queue is registered in the Gateway and added to the Known JMS Queues table in the Manage JMS Queues form. For information on using the outbound queue in a service with JMS routing, proceed to Configure and Edit the JMS Assertion.

 

 NOTE: The JMS queue settings in the SecureSpan™ Manager must match the corresponding settings in the JNDI directory in order to connect to the SecureSpan™ Gateway.

Edit a JMS Queue

  1. In the SecureSpan™ Manager, select Tasks > Manage JMS Queues from the Main Menu. The Manage JMS Queues form appears.

  2. Click [Properties]. The JMS Queue Properties form appears. Edit the queue as required. Reference the instructions in Configure a New JMS Queue for field-level information.

  3. Click [Test Settings] to test the connection between the SecureSpan™ Gateway and JMS queue. A dialog form appears. Note the diagnostic information. If a connection could not be established, double-check the queue and Gateway settings. To contact a Layer 7 Technologies Technical Support Representative, see Technical Support and Contact Information. To close the form, click [X] in the top right corner of the window.

  4. Click [Save]. The JMS Queue Properties form is closed. The revised queue is updated in the Gateway and Known JMS Queues table in the Manage JMS Queues form.

Remove a JMS Queue

 NOTE: When you remove a queue, you unregister it from the SecureSpan™ Gateway. The queue will still exist in the WebSphere® MQ server.

 

  1. In the SecureSpan™ Manager, select Tasks > Manage JMS Queues from the Main Menu. The Manage JMS Queues form appears.

  2. Select the queue that you want to remove from the Known JMS Queues table. Click [Remove].

  3. A confirmation message appears. Click [Remove].

JMS Assertion Rules

Note the following rules when using the JMS assertion for outbound routing:

  • If JMS service request messages are sent to the Gateway, then the following HTTP-related assertions cannot be used in the security policy:

    • The IP Address Range assertion from the Service Availability category of the [Assertions] tab

    • The HTTP Basic, HTTP Digest, and HTTP Client Certificate assertions from the Authentication category of the [Assertions] tab

    • The SSL Transport assertion from the Transport Layer Security (TLS) category of the [Assertions] tab.

  • If HTTP request messages are sent to the Gateway, then all assertions can be used in the security policy.

 

 NOTE: For information on the general policy and assertion rules used in the SecureSpan™ Manager, see Security Policy Organization and Rules. For information on configuring, editing, and manipulating a security policy in the Policy Development window, see Configure and Edit Security Policies.

Configure and Edit the JMS Assertion

 NOTE: Inbound JMS queues are used by published JMS services and do not require an assertion. The JMS service publication process is the same as the publication process for a regular service (see Services Overview). If publishing a service that will accept incoming messages from a JMS queue, ensure that the appropriate inbound JMS queue is monitored for the messages. See About JMS Queues for more information. JMS  messages will be processed according to the assertions defined in the security policy.

The JMS routing assertion — called "Route to JMS Queue" — appears in the Message Routing category of the [Assertions] tab in the SecureSpan™ Manager. The JMS assertion instructs the SecureSpan™ Gateway to route service request messages to a selected outbound queue configured in Configure and Edit JMS Queues.

 

 NOTE: Consult the rules in JMS Assertion Rules before using the JMS assertion.

Add and Configure the JMS Assertion

 NOTE: The following instructions assume that the SecureSpan™ Manager is open.

 

  1. Access the Policy Development window for the target service by one of two methods:

 TIP:  If not already disabled, it is strongly recommended that you disable a service while constructing its security policy to prevent unsecured and unauthorized access. A disabled service contains a red "X" through its left-side icon. See Manage Published Services for information.

    • Right-click the service name in the [Services] tab. A drop-down menu appears. Select Policy Assertions. Or,

    • Double-click the service in the [Services] tab.

  1. The Policy Development window appears. Add the JMS assertion to the security policy by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or   

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The JMS Routing Properties form appears. Select the target outbound queue from the JMS Queue drop-down list.

 NOTE: Click [New Queue] to create a new queue. The Add JMS Queue form appears. See Configure and Edit JMS Queues for instructions and more information.

  1. Click [OK]. The assertion is added to the Policy Development window. Proceed to the "Validate a Security Policy" section in Configure and Edit Security Policies.

 

 NOTE: Note the organization rules in JMS Assertion Rules and in Security Policy Organization and Rules when organizing the JMS assertion in the Policy Development window.

Edit the JMS Assertion

  1. In the Policy Development window, right-click the "Route to JMS Queue" assertion. A drop-down menu appears. Select Routing Properties.

 TIP: You can also double-click the assertion in the Policy Development window to open the JMS Routing Properties form.  

  1. The JMS Routing Properties form appears. Select a new queue from the JMS Queue drop-down list.

 NOTE: Click [New Queue] to create a new queue. The Add JMS Queue form appears. See Configure and Edit JMS Queues for instructions and more information.

  1. Click [OK]. The modified assertion is added to the Policy Development window. Proceed to the "Validate a Security Policy" section in Configure and Edit Security Policies.

 

 NOTE: Note the organization rules in JMS Assertion Rules and in Security Policy Organization and Rules when organizing the JMS assertion in the Policy Development window.
