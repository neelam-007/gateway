Manage SOAP Web Services

NOTE: The SecureSpan™ Manager differentiates between SOAP Web services and XML or non-SOAP applications. To publish or edit an XML application, see Manage XML Applications. The Publish SOAP Web Service Wizard and Create WSDL Wizard support the WSDL 1.1 standard. For information on disabling/enabling, renaming, or deleting a published Web service, see Manage Published Services.

The SecureSpan™ Manager guides you through the Web service publication process. When publishing a Web service with an existing WSDL (Web Services Description Language) document, the Publish SOAP Web Service Wizard allows you to easily enter the location, access credentials, and SecureSpan™ Gateway access option for the Web service (see Publish a SOAP Web Service). When publishing a Web service that does not have an existing WSDL document, the Create WSDL Wizard allows you to easily configure the WSDL elements that describe the business services, transactions, and electronic access instructions for the Web service (see Publish a SOAP Web Service from Created WSDL). Publication adds a Web service to the SecureSpan Manager's [Services] tab, establishes the Web service's initial policy in the Policy Development window, and allows authorized clients to access the Web service through the SecureSpan Gateway.

 

You can publish multiple instances of the same Web service when each contains a unique resolution URI (see Specify a Custom Routing URI). After publication, the SecureSpan Manager also allows you to view the Web service WSDL code and change or reset the Web service WSDL from an established WSDL document without changing its existing policy.

Publish a SOAP Web Service

NOTE: Use the Publish SOAP Web Service Wizard to publish a Web service with an existing WSDL document. When publishing a Web service that does not have a WSDL document, use the Create WSDL Wizard discussed in Publish a SOAP Web Service from Created WSDL.

In the SecureSpan™ Manager, a SOAP Web service with an existing WSDL document is published with the Publish SOAP Web Service Wizard. The Wizard contains three screens corresponding to the three main tasks in the configuration process:

  1. The Web Service URL screen locates the WSDL for the Web service.

  2. The Access Control screen allows you to define simple Web service encryption, access control, and authentication rules.

  3. The Credential Transportation screen specifies how the SecureSpan™ Gateway can gain access to the Web service.

 

You can view screen-specific information and instructions in the Tips Box located near the bottom of the Publish SOAP Web Service Wizard.

  1. Access the Publish SOAP Web Service Wizard by one of three methods:

    • Click the "Publish SOAP Web Service" link on the Home Page

    • Click [Tasks] > Publish SOAP Web Service from the Main Menu

    • Right-click the "Services @ machinename.domain.com" title at the top of the [Services] tab. A drop-down menu appears. Select Publish SOAP Web Service.

  1. The Web Service URL screen of the Publish SOAP Web Service Wizard appears. Define the WSDL document for the Web service by one of the following methods:

Enter the WSDL URL

If you have a WSDL URL for the Web service:

    1.  Enter the Web service URL into the URL field. Click [Next].

    2. One of the following will occur:

      • A pop-up error message notifies you that the WSDL URL could not be resolved by the SecureSpan Gateway. Do the following:

        1. Note the errors. Click [OK] to close the dialog box.

        2. Re-enter the WSDL URL into the URL field. Click [Next].

        3. Continue to fix and resolve the URL until the WSDL URL is entered in the URL field and the Access Control screen of the Publish SOAP Web Service Wizard appears, and then proceed to Step #3. If the WSDL URL cannot be resolved, see Technical Support to contact a Technical Support Representative.

      • A pop-up message appears while the WSDL URL is being resolved. The message automatically disappears, the WSDL URL is entered in the URL field, and the Access Control screen of the Publish SOAP Web Service Wizard appears. Proceed to Step #3.

Extract the WSDL URL from a WSIL URL

To extract the WSDL URL for the Web service from a WSIL (Web Services Inspection Language) URL:

    1. Enter the WSIL URL into the URL field. Click [Next].

    2. The Select WSDL Target from WSIL List form appears. The Web services registered in the WSIL document appear in the table by Web service name and WSDL URL. Select the target Web service.

    3. Click [OK]. One of the following will occur:

      • A pop-up error message notifies you that the WSDL URL could not be extracted from the WSIL URL or that the selected WSDL URL could not be resolved by the SecureSpan Gateway. Do the following:

        1. Note the errors. Click [OK] to close the dialog box.

        2. Repeat Steps #1-2 above. Click [Next].

        3. Continue to fix and resolve the URL until the selected WSDL URL is entered in the URL field and the Access Control screen of the Publish SOAP Web Service Wizard appears, and then proceed to Step #3. If the WSIL URL or WSDL URL errors continue, see Technical Support to contact a Technical Support Representative.

      • A pop-up message appears while the WSDL URL is being resolved. The message automatically disappears, the WSDL URL is entered in the URL field, and the Access Control screen of the Publish SOAP Web Service Wizard appears. Proceed to Step #3.

Search the UDDI for the WSDL or WSIL URL

WARNING: In order to use this method for obtaining a WSDL or WSIL URL, you must have the Systinet UDDI Registry product (www.systinet.com) and have at least one Systinet UDDI URL configured in the SecureSpan™ Gateway. The Systinet UDDI URLs are configured in the SecureSpan Gateway's "/ssg/etc/conf/uddi.properties" file. Systinet provides a set of APIs that, via the SecureSpan Gateway, can be used by a client application to retrieve information about Web services registered in the Systinet UDDI Registry server.

To search the Systinet UDDI (Universal Description, Discovery, and Integration) Registry for the Web service:

    1. Click [UDDI]. The Search WSDL form appears.

    2. Select a Systinet UDDI URL from the UDDI Registry URL drop-down list.

NOTE: The URL(s) in the UDDI Registry URL drop-down list are the Systinet UDDI URLs configured in the SecureSpan™ Gateway.

    1. Enter criteria for searching for the target Web service into the Service Name field. Click [Search].

TIP: Check the Case Sensitive check box to restrict the search results to the case of the criteria entered in the Service Name field.

    1. The Web services matching the search criteria appear in the table by Web service name and WSDL URL or WSIL URL. Select the target Web service.

    2. Click [OK]. One of the following will occur:

Selected URL is a WSDL URL

If the selected URL was a WSDL URL:

      • A pop-up error message notifies you that the WSDL URL could not be resolved by the SecureSpan Gateway. Do the following:

        1. Note the errors. Click [OK] to close the dialog box.

        2. Repeat Steps #1-4 above. Click [Next].

        3. Continue to fix and resolve the URL until the selected WSDL URL is entered in the URL field and the Access Control screen of the Publish SOAP Web Service Wizard appears, and then proceed to Step #3. If the WSDL URL errors continue, see Technical Support to contact a Technical Support Representative.

      • A pop-up message appears while the WSDL URL is being resolved. The message automatically disappears, the WSDL URL is entered in the URL field, and the Access Control screen of the Publish SOAP Web Service Wizard appears. Proceed to Step #3.

Selected URL is a WSIL URL

If the selected URL is a WSIL URL, then follow the instructions in Extract the WSDL URL from a WSIL URL above to extract the target Web service, or WSDL URL, from the WSIL document.

 

NOTE: The SecureSpan™ Manager checks for duplicity during the publication process and will display an error message and prompt you for a URI (see Specify a Custom Routing URI) if the Web service WSDL is not unique. Upon publication, the WSDL URL in the URL field appears as an HTTP(S) Routing assertion in the Web service's initial policy (see Manage Policies and Configure and Edit Message Routing Assertions) and the name of the Web service is extracted from the resolved WSDL and added to the [Services] tab.

 

  1. In the Access Control screen:

    1. Optionally check the Require SSL/TLS Encryption check box to require that all requestors consume the Web service through the SSL entry point.

    2. Select one of two access control options:

      • Click Allow Anonymous Access to permit requestors to access the Web service anonymously (without credentials)

      • Click Require Users to Authenticate to require that requestors provide credentials to gain Web service access. Define the authentication details for this option as follows:

NOTE: If you specify an authentication method and one or more authorized identities in the Publish SOAP Web Service Wizard, then Web service access will be restricted to those identities. The initial policy will automatically be populated with an authentication assertion for each Grant Access to Users/Groups assertion corresponding to each selected identity. See Configure and Edit Access Control Assertions for more information.

        1. Select an authentication method from the Authentication Method drop-down list. The authentication method determines what information users and groups are required to provide to gain Web service access.

        2. Select an identity provider that contains the authorized users and groups from the Identity Provider drop-down list.

NOTE: You can give LDAP Identity Provider, Federated Identity Provider, and Internal Identity Provider (see Manage Identity Providers) users and groups access to the Web service.

        1. Authorize users and groups to use the Web service by moving them from the No Permission window to the Have Permission window as follows:

          • Click [Add All] to authorize all users and groups in the No Permission window by moving them to the Have Permission window

          • Authorize one or more selected users and groups as follows:

            • To authorize a single user or group, click the user or group name and click [Add]

            • To authorize a continuous range of users and/or groups, hold down your left mouse button as you move from the first user or group to the last user or group in your target range. Or, left-click the first user or group in the range, hold down the [Shift] button on your keyboard, and then left-click the last user or group in the range. Click [Add]

            • To authorize multiple users or groups placed throughout the list, hold down the [Ctrl] button on your keyboard as you click each user or group name. Click [Add].

NOTE: To de-authorize users and/or groups, move them from the Have Permission window to the No Permission window. Select one or more users and/or groups as outlined above, then click the [Remove] button to remove them from the Have Permission window. To remove all users and/or groups from the Have Permission window, click [Remove All].

        1. Optional: If you need to authorize users or groups from another identity provider, select the new identity provider name from the Identity Provider drop-down list and repeat the instructions in Step #3.

 

  1. Click [Next]. The Credential Transportation screen appears:

    1. By default, the Web Service URL field is populated with the Web service access information provided in the WSDL document. To change the URL where the SecureSpan Gateway can access the Web service:

      1. Click [Change].

      2. Enter a new URL in the Web Service URL field.

      3. Click [Default].

    1. Select one of two SecureSpan Gateway access control options:

      • Click The SecureSpan Gateway can access this protected Web service anonymously to instruct the SecureSpan Gateway to access the protected Web service without authentication

      • Click The SecureSpan Gateway will need to provide credentials to access this Web service to instruct the SecureSpan Gateway to provide credentials when connecting to the Web service.

 

  1. Click [Finish]. The Web service is immediately added to the [Services] tab.

 

TIP: Disable the published Web service until its policy is completed. See Manage Published Services for instructions. For information on policies, see Manage Policies.

 

Publish a SOAP Web Service from Created WSDL

NOTE: The SecureSpan™ Manager's Create WSDL Wizard should only be used by advanced users who are familiar with WSDL, XML, SOAP, and the SOAP protocols. If a WSDL document exists for the Web service you want to publish, use the Publish SOAP Web Service Wizard discussed in Publish a SOAP Web Service instead.

The SecureSpan™ Manager guides advanced users through the WSDL definition process. The Create WSDL Wizard contains six screens corresponding to the six main and child elements in the WSDL document:

  • The Overview screen describes the overall process and information required by the Create WSDL Wizard

  • The Definition screen configures the root definitions element and its child elements. These elements define the particulars of the Web service

  • The Messages screen configures the messages element. A Web service contains multiple messages with one or more logical parts that define the communication between the Web service client and server

  • The Port Type and Operations screen configures the port type element that includes the set of operations used in the Web service. Each operation refers to one input message and one output message configured in the previous Messages screen

  • The Port Type Binding screen configures the binding element that specifies the binding definitions that govern message formatting and protocol details

  • The Services screen configures the service element that defines the Web service endpoint address (URL) and access port.

 

TIP: Click [Preview] at any time during the configuration process to view the read-only WSDL for the Web service. To close the WSDL form, click the [X] in the top right corner of the form window.

 

  1. Access the Create WSDL Wizard by one of three methods:

    • Click the "Create WSDL" link on the Home Page

    • Click [Tasks] > Create WSDL from the Main Menu

    • Right-click the "Services@https://servername:port" item in the [Services] tab. A drop-down menu appears. Select Create WSDL.

  1. The Overview screen of the Create WSDL Wizard appears. Read the information and instructions. Click [Next].

  2. The Definition screen appears:

    1. Enter a descriptive name for the Web service into the Name field. For example, "Warehouse."

NOTE: The name entered in the Name field will appear in the [Services] tab.

    1. By default, an "http://tempuri.org/Name.wsdl" URI is placed in the Target Namespace field. Replace the default namespace with a namespace that relates to your Web service application. This namespace can be a URI or a URN.

NOTE: The namespace plus SOAP Action combination determines the uniqueness of the Web service. The SecureSpan™ Manager checks for duplicity during the publication process and will display an error message and prompt you for a URI (see Specify a Custom Routing URI) if the Web service WSDL is not unique. A namespace that looks like a URL does not necessarily point to a Web-based resource. Avoid the use of relative URI namespaces. A URI namespace causes XML canonicalization problems and prevents the use of message-level security such as the signature and encryption assertions in the XML Security category in the [Assertions] tab (see Configure and Edit XML Security Assertions). Upon publication, the URL in the URL field will appear as an HTTP(S) Routing assertion in the Web service's initial policy. See Manage Policies and Configure and Edit Message Routing Assertions for more information.

    1. By default, an "http://schemas.xmlsoap.org/wsdl/" URI is placed in the Default Namespace field. The URI is the namespace convention for the main elements in the WSDL document.

    2. The Namespace Details window displays read-only information about the namespaces.

 

  1. Click [Next]. The Messages screen appears:

    1. In the Message List section of the screen, click [Add]. A new message name appears in the Message List window. Double-click and highlight the default name and replace it with a name that describes the message type. For example, "GetQuoteResponse."

NOTE: To remove a message from the Message List window, select the message name and click [Remove].

    1. In the Message Part List section of the screen, click [Add]. A new part item appears in the Message Part List window:

      1. In the Name column of the Message Part List window, double-click and highlight the default name and replace it with a name that describes the message part. For example, "stockSymbol."

      2. In the Type column of the Message Part List window, click and select the specific part type from the drop-down list.

      3. To add additional parts to the message selected in the Message List window, click [Add]. Define the part name and type as outlined above.

NOTE: To remove a part from the Message Part List window, select the part name and click [Remove].

    1. In the Message List section of the screen, click [Add] to add another message to the Message List window. Repeat Step #2 above to define one or more parts for the message. Repeat as necessary to add additional messages and parts.

 

  1. Click [Next]. The Port Type and Operations screen appears:

    1. Enter a descriptive name for the port type into the Name field. For example, "StockInfo."

    2. In the Operation List section of the screen, click [Add]. A new operation appears in the Operation List window:

      1. In the Name column of the Operation List window, double-click and highlight the default name and replace it with a name that describes the operation. For example, "getQuote."

      2. In the Input column of the Operation List window, click and select an input message from the drop-down list. For example, "getQuoteRequest."

      3. In the Output column of the Operation List window, click and select an output message from the drop-down list. For example, "getQuoteResponse."

NOTE: An operation consists of an input and output message. The messages in the drop-down lists were defined in the Messages screen of the Create WSDL Wizard.

      1. Click [Add] to add additional operations to the Operation List window. Name and define messages for the operation(s) as outlined above.

NOTE: To remove an operation from the Operation List window, select the operation name and click [Remove].

 

  1. Click [Next]. The Port Type Binding screen appears:

    1. Enter a descriptive name for the port type binding into the Name field. For example, "StockServiceSOAPBinding."

    2. Select a message format for the operations from the Style drop-down list.

    3. The read-only default namespace URI displays in the Transport field. The URI is the namespace of the transport-specific elements in the WSDL document.

    4. In the Operations window:

NOTE: The Operations window lists the operations defined in the Port Type and Operations screen of the Create WSDL Wizard.

      • Select an operation. Double-click and highlight the attribute in the corresponding SOAP Action column. If necessary, replace the default attribute name

      • Repeat for all operations in the Operations window.

 

  1. Click [Next]. The Service screen appears:

    1. Enter a descriptive name for the service element into the Name field. For example, "StockQuote."

    2. A default access port for the Web service appears in the Port field. Do not edit the default value.

    3. By default, the read-only port type binding name appears in the Binding field.

TIP: To update the default URL in the Address field with the information entered in the Name and Port fields, click [Back] and then [Next] in the Create WSDL Wizard. The port type binding name in the Binding field was entered in the Port Type Binding screen of the Create WSDL Wizard.

    1. By default, an "http://localhost:8080/ws/Name" URL is placed in the Address field. The "localhost" is the host name of the server hosting the Web service, "8080" is the TCP port at which the Web service can be reached, "ws/" is a sample folder that might contain the Web service, and "Name" is the Web service name that was entered in the Name field. Replace one or more values as required to construct a valid URL for the Web service.

NOTE: The URL in the Address field will determine the default Web service message routing assertion (see Configure and Edit Message Routing Assertions) in the policy.

 

  1. Optional: Click [Preview] to view the read-only WSDL for the Web service. To close the WSDL form, click the [X] in the top right corner of the form window.

TIP: Since you cannot edit a Web service once it is published, it is recommended that you use the Create WSDL Wizard's [Back] and [Next] buttons to scroll through and double-check each screen before you click [Finish].

  1. Click [Finish]. The disabled Web service is immediately added to the [Services] tab. Construct a policy for the Web service as outlined in Manage Policies, then enable the Web service as outlined in Manage Published Services.

 

Specify a Custom Routing URI

The SecureSpan™ Manager allows you to specify a custom routing URI for a Web service. The routing URI property of a Web service allows the SecureSpan™ Gateway to resolve an incoming request to a specific URL. For example, if a Web service has a custom routing URI value of "/primary", then SOAP requests meant for that published Web service must be sent to the SecureSpan Gateway at the URL "http://gatewayhostname:httpport/primary" or "https://gatewayhostname:httpsport/primary".

 

When a custom routing URI value is not provided, the SecureSpan Gateway receives requests at the default incoming URL (such as "http://gatewayhostname:httpport") and uses the message URN and soapaction values to resolve the destination Web service. Since the default service resolution process uses URN and soapaction values, customizing the routing URI value of a published Web service allows for the same WSDL to be published more than once. Without this distinction, duplication would result in runtime ambiguity. When a custom URI is set for a Web service, the Web service name in the [Services] tab contains the custom URI as a suffix.

 

A custom URI is assigned to a Web service using the Web Service Routing URI form. The Web Service Routing URI form will automatically appear if ambiguity is detected during the Web service publication process (for example, if a WSDL is provided that was already used to publish another Web service).

 

TIP: If using the Web Service Routing URI to assign a custom URI to a planned duplicate Web service, assign a unique URI to the original published Web service first, and then publish the duplicate using the Publish SOAP Web Service Wizard (see Publish a SOAP Web Service). The name of the original Web service in the [Services] tab will change to incorporate the routing URI. For example, for the final routing URL/URI combination "http://webservice:port/primary", the original service name "Web Service" becomes "Web Service [/primary]".

 

  1. Right-click the published Web service in the [Services] tab. A drop-down menu appears.

NOTE: The Web Service Routing URI form will automatically appear if ambiguity is detected during the Web service publication process.

  1. Select View Gateway URL. The Web Service Routing URI form appears.

  2. Select the Custom routing URI option. In the field, enter a unique routing URI for the Web service. For example, "primary."

  3. Click [OK]. The node name of the published Web service in the [Services] tab changes to incorporate the routing URI. For example, "Web Service" will now be referenced as "Web Service [/primary]."

 

View Web Service WSDL

View and search the WSDL document for a published Web service as follows:

  1. Right-click the Web service name in the [Services] tab. A drop-down menu appears.

NOTE: The WSDL in the WSDL form is read-only.

  1. Select View WSDL. The WSDL form appears.

  1. To optionally search the WSDL document nodes or copy the WSDL to an external location:

Search WSDL Nodes

To search the WSDL nodes:

    1. Right-click inside the WSDL form. A drop-down menu appears.

    2. Select Search a Node. The Search form appears containing an expandable list of the nodes in the WSDL document. To view the location of a node in the Search form, click the node. The node's corresponding location in the WSDL form is highlighted.

    3. Click [OK] to close the Search form.

Copy WSDL

NOTE: Since the WSDL in the WSDL form is read-only, you can only copy content to an external location. For example, you cannot paste a copied WSDL element back into the WSDL form. To change the WSDL for a published Web service, you must reset the document as outlined in Reset Web Service WSDL.

To copy selected WSDL form content to an external location:

    1. Select the target content.

    2. Right-click inside the WSDL form. A drop-down menu appears. Select Copy.

    3. Paste the content to a location outside of the WSDL form.

  1. To close the WSDL form, click the [X] in the top right corner of the form window.

 

Reset Web Service WSDL

NOTE: You will need a WSDL document to reset or change the WSDL for an existing Web service.

The SecureSpan™ Manager allows you to reset or change the WSDL for a published Web service with another WSDL document. Once reset, the existing policy will become active for the resolution parameters extracted from the new WSDL document.

  1. Right-click the Web service name in the [Services] tab. A drop-down menu appears.

  1. Select Reset WSDL. The Reset WSDL form appears.

  1. In the field, enter the URL that will resolve the new Web service WSDL. Click [OK]. The routing assertion for the policy (see Configure and Edit Message Routing Assertions) is updated with the new WSDL URL.
