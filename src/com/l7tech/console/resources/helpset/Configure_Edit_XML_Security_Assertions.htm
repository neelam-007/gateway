Sign Request Element

NOTE: The Sign Request Element assertion supports the W3C XML Signature 1.0 standard. The assertion can only be used in a Web service policy. You can add a Sign Request Element assertion for each element of the request message that you want signed. In a policy, the Sign Request Element assertion must be placed anywhere before the routing assertion (see Configure and Edit Message Routing Assertions).

The Sign Request Element assertion allows you to require that the Web service requestor sign a specific element whenever it appears in the request message. The assertion checks if a selected XPath expression matches the corresponding element in the request message. If a matching element is not found, then nothing in the request message must be signed. If a matching element is found, then that element must be signed. A matching element is considered to be signed if the signature covers at least the entire element including opening and closing tags, attributes, and all child content.

 

WARNING: A WSS Signature assertion, WS Secure Conversation assertion, or SAML Assertion (see Configure and Edit Access Control Assertions) must be added for each Sign Request Element assertion in a policy.

 

Add and Configure the Sign Request Element Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The "Sign request element" assertion is added to the Policy Development window. Right-click the assertion. A drop-down menu appears. Select Sign Request Element Properties.

  1. The Sign Request Element Properties form appears. Select the operation that contains the element that you want signed from the Web Service Operations window.

NOTE: The operations are loaded from the Web service WSDL code. For information about the WSDL that defines a Web service, see Manage SOAP Web Services.

  1. The XML code for the operation appears as expanded nodes in the Code window. Click the target element.

NOTE: Clicking, or highlighting, an element selects it (and any child code) for the assertion signature requirement.

  1. Click [OK]. Assertion changes take effect immediately.

 

OPTION: The recipient of the Sign Request Element assertion can be changed using the Change WSS Recipient form (see Change the WSS Assertion Recipient).

 

Edit the Sign Request Element Assertion

  • In the Policy Development window, right-click the "Sign request element" assertion. A drop-down menu appears. Select Sign Request Element Properties

  • The Sign Request Element Properties form appears. Edit the assertion properties as outlined in Add and Configure the Sign Request Element Assertion.

 

Encrypt Request Element

NOTE: The Encrypt Request Element assertion supports the W3C XML Encryption 1.0 standard. The assertion can only be used in a Web service policy. You can add an Encrypt Request Element assertion for each element of the request message that you want encrypted. In a policy, the Encrypt Request Element assertion must be placed anywhere before the routing assertion (see Configure and Edit Message Routing Assertions).

The Encrypt Request Element assertion allows you to require that the Web service requestor encrypt a specific element whenever it appears in the request message. The assertion checks if a selected XPath expression matches the corresponding element in the request message. If non-empty matches exist but were not encrypted by the sender, then the assertion fails; otherwise, it succeeds. A matching element is considered to be encrypted if it is completely encrypted or if all of its child content (except leading and trailing white space) is encrypted. Elements that are empty or contain only white space can be left un-encrypted in the request message.

Add and Configure the Encrypt Request Element Assertion

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  1. The "Encrypt request element" assertion is added to the Policy Development window. Right-click the assertion. A drop-down menu appears. Select Encrypt Request Element Properties.

  1. The Encrypt Request Element Properties form appears. Select the operation that contains the element that you want encrypted from the Web Service Operations window.

NOTE: The operations are loaded from the Web service WSDL code. For information about the WSDL that defines a Web service, see Manage SOAP Web Services. The SecureSpan™ Manager will not allow you to encrypt the "/soapenv:Envelope" element in the Encrypt Request Element Properties form. You can, however, encrypt a child element within the envelope such as "/soapenv:Envelope/soapenv:Body."

  1. The XML code for the operation appears as expanded nodes in the Code window. Click the target element.

TIP: A matching element's own opening and closing tags and tag attributes do not have to be encrypted. To force the encryption of an entire element — including opening and closing tags, attributes, and white space content — match the XPath expression to the parent element of the message. Clicking, or highlighting, an element selects it (and any child code) for the assertion encryption requirement.

  1. Click [OK]. Assertion changes take effect immediately.

 

OPTION: The recipient of the Encrypt Request Element assertion can be changed using the Change WSS Recipient form (see Change the WSS Assertion Recipient).

 

Edit the Encrypt Request Element Assertion

  • In the Policy Development window, right-click the "Encrypt request element" assertion. A drop-down menu appears. Select Encrypt Request Element Properties

  • The Encrypt Request Element Properties form appears. Edit the assertion properties as outlined in Add and Configure the Encrypt Request Element Assertion.

 

Sign Response Element

NOTE: The Sign Response Element assertion supports the W3C XML Signature 1.0 standard. The assertion can only be used in a Web service policy. You can add a Sign Response Element assertion for each element of the response message that you want signed. In a policy, the Sign Response Element assertion must be placed anywhere after the routing assertion (see Configure and Edit Message Routing Assertions).

The Sign Response Element assertion allows you to require that the SecureSpan™ Gateway sign a specific element whenever it appears in the response message before sending it back to the Web service requestor. The assertion checks if a selected XPath expression matches the corresponding element in the response message. A match is signed by the SecureSpan Gateway as long as a SOAP response is available and the assertion XPath is valid. In the SecureSpan™ Bridge, the assertion only succeeds if the corresponding response elements were signed by the SecureSpan Gateway. Unsigned XPath matches fail while signed matches pass. A matching element is considered to be signed if the signature covers at least the entire element including opening and closing tags, attributes, and all child content.

  1. The "Sign response element" assertion is added to the Policy Development window. Right-click the assertion. A drop-down menu appears. Select Sign Response Element Properties.

  1. The Sign Response Element Properties form appears. Select the operation that contains the element that you want signed from the Web Service Operations window.

NOTE: The operations are loaded from the Web service WSDL code. For information about the WSDL that defines a Web service, see Manage SOAP Web Services.

  1. The XML code for the operation appears as expanded nodes in the Code window. Click the target element.

NOTE: Clicking, or highlighting, an element selects it (and any child code) for the assertion signature requirement.

  1. Click [OK]. Assertion changes take effect immediately.

You can add an Encrypt Response Element assertion for each element of the response message that you want encrypted. In a policy, the Encrypt Response Element assertion must be placed anywhere after the routing assertion (see Configure and Edit Message Routing Assertions).

The Encrypt Response Element assertion allows you to require that the SecureSpan™ Gateway encrypt a specific element whenever it appears in the response message before sending it back to the Web service requestor. The assertion checks if a selected XPath expression matches the corresponding element in the response message. If any non-empty matches exist, their child content will be encrypted by the SecureSpan Gateway as long as a SOAP response is available and the XPath is valid. In the SecureSpan™ Bridge, the assertion only succeeds if the corresponding response element's child content was encrypted by the SecureSpan Gateway.

WSS Replay Protection

The WSS Replay Protection assertion allows you to require that the Web service request message include a valid, unique, and signed timestamp. If the SecureSpan™ Gateway detects a duplicate creation timestamp in a message, if an expired timestamp is present and has already passed, or if the creation timestamp is more than 30 days old, then the message will be rejected as a possible replay.

 

TIP: Since the system has to keep track of all timestamps created over the past 30 days, using the WSS Replay Protection assertion in a SecureSpan™ Gateway cluster can increase request message processing time and use more memory. Place the WSS Replay Protection assertion after a Grant Access to Users/Groups assertion (see Configure and Edit Access Control Assertions) to help confine the protection to successfully authenticated messages, thereby reducing system processing and memory requirements.

 

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.
