Configure and Edit Message Routing Assertions

 NOTE: The JMS Routing assertion in the Message Routing category is an optional assertion that is disabled in standard versions of the SecureSpan™ Solution. See Optional Add-On Assertions Overview for more information.

In the SecureSpan™ Manager, two assertions are available in the Message Routing category of the [Assertions] tab: An optional JMS Routing assertion that is addressed in Optional Add-On Assertions Overview, and an HTTP(S) Routing assertion discussed below:

 

The Message Routing Category of the [Assertions] Tab

HTTP(S) Routing

 NOTE: The HTTP(S) Routing assertion supports the HTTP 1.1 standard.

The HTTP(S) Routing assertion defines where a Web service or XML application message is sent and what authentication credentials it uses. A message routing assertion is an essential security policy element. When you publish a service using the Publish Web Service Wizard, Create WSDL Wizard, or Publish XML Application Wizard, the SecureSpan™ Manager automatically adds the service URL specified during the publication process as an HTTP(S) Routing assertion in the published service's initial security policy. For information on publishing a service, see Services Overview.

Configure the HTTP(S) Routing Assertion

 NOTE: The HTTP(S) Routing assertion should already be present in the Policy Development window. If not, then add the assertion by one of the following methods: 1. Drag and drop the assertion from the [Assertions] tab into the Policy Development window or, 2. Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

 

  1. In the Policy Development window, right-click the "Route to http" assertion. A drop-down menu appears. Select Routing Properties.

 TIP: You can also double-click the assertion in the Policy Development window to open the HTTP(S) Routing Properties form.   

  1. The HTTP(S) Routing Properties form appears. If required, edit the URL path to the service in the Service URL field.

 TIP: Click [Default] to populate the Service URL field with the URL specified during the service publication process.

  1. In the Service Authentication section of the form, optionally configure one of the following authentication options for the service:

 NOTE: No credentials are required for an anonymous service.

Identity Authentication

To enable HTTP basic authentication to a protected resource back-end:

 

 NOTE: For more information about the HTTP Basic authentication assertion, see Configure and Edit Authentication Assertions.

 

    1. Select the Identity Authentication option.

    2. Enter the user name into the User Name field.

    3. Enter the user password into the Password field.

    4. Enter the authentication realm into the Realm field. The realm can be a secure virtual host or a directory.

    5. Check the optional TAI Identity Pass check box to provide a Trust Association Interceptor (TAI) third-party authentication pass.

 NOTE: The TAI Identity Pass option configures the SecureSpan™ Gateway to include an IV_USER=username tag in an authenticated outgoing back-end request as both an HTTP cookie and as an HTTP header. In the tag, the "username" is the user name of the authenticated user.

    1. Proceed to Step #4.

SAML Security

 NOTE: The SAML Security option is only enabled for Web service security policies.

The HTTP(S) Routing assertion can optionally attach a SAML sender-vouches ticket to each outgoing back-end request that was authenticated by the SecureSpan™ Gateway. This ticket contains the user name of the authenticated user along with an expiry time, and is signed by the Gateway using the Gateway's SSL certificate. To enable SAML authentication in the HTTP(S) Routing assertion:

 

 NOTE: The SAML security option in the HTTP(S) Routing assertion differs from the SAML Security assertion in the Authentication category of the [Assertions] tab. The SAML Security authentication assertion is used in an identity bridging configuration to require that the incoming front-end request be signed and include a SAML holder-of-key ticket. See Configure and Edit Authentication Assertions and Identity Bridging Workflow Using SAML for more information.

 

    1. Select the SAML Security option.

    2. Optional: Enter or select the time limit for the SAML ticket in the Ticket Expiry (in minutes) field.

    3. Proceed to Step #4.

 

  1. Click [OK]. The configured assertion is added to the Policy Development window.

Edit the HTTP(S) Routing Assertion

  • In the Policy Development window, right-click the "Route to http" assertion. A drop-down menu appears. Select Routing Properties.

 TIP: You can also double-click the assertion in the Policy Development window to open the HTTP(S) Routing Properties form.   

  • The HTTP(S) Routing Properties form appears. Edit the properties as outlined in Steps #2-4 in Configure the HTTP(S) Routing Assertion.
