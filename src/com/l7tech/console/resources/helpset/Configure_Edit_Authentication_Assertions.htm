In the SecureSpan™ Manager, the following assertions are available in the Authentication category of the [Assertions] tab:

NOTE: Multiple authentication methods may be required when constructing a security policy for merged organizations. In this case, a single Web service may have to respond to two sets of users: one within the parent organization that requires one authentication method, and the other within the acquired organization that requires a different authentication method. The SecureSpan™ Manager allows you to customize a Web service security policy to suit the needs of any type of organization. For general policy and assertion information and rules, see Security Policies Overview and Rules.

Grant Access to Users and Groups

The Grant Access to Users and Groups assertion allows you to provide Web service access to particular External Identity Provider (EIP) and Internal Identity Provider (IIP) groups and/or users.

NOTE: A security policy with more than one Grant Access to Users and Groups assertion must organize the identities into an "At least one assertion must evaluate to true" assertion folder. To add a folder to the Policy Development window, right-click any assertion and select "Add 'One or More' Assertion" from the drop-down menu. Drag and drop configured groups and/or users into the folder as required. For more information, see Security Policies Overview and Rules and Configure and Edit Security Policies.

  1. Add the assertion to the Policy Development window by one of three methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Move Assertion) button in the Assertion Tool Bar

    • Right-click an "All assertions must evaluate to true" or "At least one assertion must evaluate to true" assertion folder. A drop-down menu appears. Select Add User or Group.

  1. The Search Identity Provider form appears. Select the EIP or IIP name that contains the target group(s) and/or user(s) from the Search drop-down list.

NOTE: To initiate a new search in the Search Identity Provider form, click [New Search]. To stop a search that is in progress, click [Stop].

  1. Select the search target — groups, users, or both — from the Type drop-down list.

  2. If necessary, specify any search criteria using the Name drop-down list and field.

NOTE: The Search Identity Provider form supports the following search characters in the Name field:

  • The asterisk truncation operator (*) retrieves terms with a common initial spelling. It substitutes a string of zero or more characters when searching the Internal and External Identity Provider

  • The question mark wild card operator (?) substitutes zero or one character in a term. Use within a word or at the end of a word when searching the Internal Identity Provider.

  1. Click [Search]. Search results appear in the Search Results window. Select one or more groups and/or users as follows:

NOTE: A single Grant Access to Users and Groups assertion will be added to the security policy for each group and/or user selected in the Search Identity Provider form.

Select a Single Group or User

To select a single group or user for the security policy, click the group or user's name and click [Select]. The Search Identity Provider form closes and an assertion for the group or user is added to the Policy Development window.

Select Multiple Groups and/or Users

    1. Select multiple groups or users for the security policy by one of two methods:

      • To select a consecutive range of groups or users, hold down your left mouse button as move from the first group or user to the last group or user in your target range. Or, press [Shift] on your keyboard, left-click the first group or user in the range, and then left-click the last group or user in the range. Click [Select]

      • To add groups or users placed throughout the list, press [Ctrl] on your keyboard as you click each group or user name that you want to add. Click [Select].

    1. The Search Identity Provider form closes and an assertion for each group or user is added to the Policy Development window. Repeat the process as required.

NOTE: You cannot edit the Grant Access to Users and Groups assertion(s). To remove an assertion, right-click the assertion name. A drop-down menu appears. Click [Delete]. To view policy assertions by the configured groups and/or users in the policy (read-only), click the [Identity View] button on the Policy Tool Bar. To switch back to the Policy Development window, click the [Policy View] button.

HTTP Basic

The HTTP Basic assertion allows you to require basic HTTP authentication — username, password, and the authentication realm as a string in the request headers — for the Web service.

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Move Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

NOTE: You cannot edit the HTTP Basic assertion. To remove an assertion, right-click the assertion name. A drop-down menu appears. Click [Delete].

HTTP Digest

The HTTP Digest assertion allows you to require digest HTTP authentication — username, password, and the authentication realm in the request headers of a "SHA" digest form — for the Web service. HTTP Digest authentication provides better security because it does not transmit the password used to authenticate.

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Move Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

NOTE: You cannot edit the HTTP Digest assertion. To remove an assertion, right-click the assertion name. A drop-down menu appears. Click [Delete].

HTTP Client Certificate

NOTE: If using the HTTP Client Certificate assertion in a security policy, you must also select the SSL Transport assertion from the Transport Layer Security (TLS) category in the [Assertions] tab. See Configure and Edit Transport Layer Security (TLS) Assertions for information.

The HTTP Client Certificate assertion allows you to require a client certificate signed by the SecureSpan™ Gateway certificate authority as part of the SSL-TLS handshake. The client certificate can later be used to authenticate the Web service requestor.

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Move Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

NOTE: You cannot edit the HTTP Client Certificate assertion. To remove an assertion, right-click the assertion name. A drop-down menu appears. Click [Delete].

WS Token Basic

NOTE: Since the WS Token Basic assertion requires the insertion of the password into the XML Web service message, it is recommended that you also select the SSL Transport assertion when using basic WS Token authentication. See Configure and Edit Transport Layer Security (TLS) Assertions for information about the SSL Transport assertion.

The WS Token Basic assertion allows you to require basic WS Token authentication — username, password, and the authentication realm in a section of the XML message — for the Web service.

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Move Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

NOTE: You cannot edit the WS Token Basic assertion. To remove an assertion, right-click the assertion name. A drop-down menu appears. Click [Delete].

XML Digital Signature

NOTE: You must require the signature of the entire XML message in the XML Security Properties form in order to use the XML Digital Signature assertion for authentication. If you sign and/or encrypt only certain message elements, then a secondary authentication method will be required in the security policy.

The XML Digital Signature assertion allows you to require that the Web service requestor digitally sign the message when making a request.

  1. Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Move Assertion) button in the Assertion Tool Bar.

  1. The assertion is added to the Policy Development window. Right-click the assertion. A drop-down menu appears. Select XML Security Properties.

TIP: You can also double-click the assertion in the Policy Development window to open the XML Security Properties form.  

  1. The XML Security Properties form appears. Confirm the selection of the Sign/encrypt the entire message option in the Signature and Encryption Scope section.

  2. Click [OK]. The assertion is added to the Policy Development window.

NOTE: If you change the signature option in the XML Security Properties form, then you will not be able to use the XML Digital Signature assertion for authentication in the security policy. The title of the XML Digital Signature assertion in the Policy Development window is the same as the title of the Sign/Encrypt Request assertion from the XML Security category of the [Assertions] tab. See Configure and Edit XML Security Assertions for more information on the Sign/Encrypt Request assertion.
