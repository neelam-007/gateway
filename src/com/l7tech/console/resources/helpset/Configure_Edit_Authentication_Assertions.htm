Grant Access to Users/Groups

 NOTE: In the SecureSpan™ Manager, the term "identity" includes both users and groups, and the term "user" represents an individual human or machine. If an existing security policy contains a user or group that has since been removed from its corresponding identity provider, then the SecureSpan™ Manager will fail the security policy during the final validation process. See the "Validate a Security Policy" section in Configure and Edit Security Policies for more information.

The Grant Access to Users/Groups assertion allows you to provide service access to particular External Identity Provider (EIP), Federated Identity Provider (FIP), and Internal Identity Provider (IIP) users and/or groups.

 

 NOTE: You cannot use the HTTP Digest authentication assertion with External Identity Provider (EIP) users or groups. For more information about the HTTP Digest assertion, see HTTP Digest. For more information about EIPs, see Add and Edit External Identity Providers.

 

  1. Add the assertion to the Policy Development window by one of three methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar

    • Right-click an "All assertions must evaluate to true" or "At least one assertion must evaluate to true" assertion folder. A drop-down menu appears. Select Add User or Group.

  1. The Search Identity Provider form appears. Select the EIP, FIP, or IIP name that contains the target user and/or group from the Search drop-down list.

 NOTE: To initiate a new search in the Search Identity Provider form, click [New Search]. To stop a search that is in progress, click [Stop].

  1. Select the search target — users, groups, or both — from the Type drop-down list.

  2. If necessary, specify any search criteria using the Name drop-down list and field.

 TIP: The Search Identity Provider form supports the following wild card characters in the Name field:

  • The asterisk wild card (*) retrieves terms with a common initial or subsequent spelling. It matches any string of zero or more characters when searching the External Identity Provider, Federated Identity Provider, and Internal Identity Provider

  • The question mark wild card (?) substitutes one character in a term. Use within a word or at the end of a word when searching the Internal Identity Provider.

  1. Click [Search]. Search results appear in the Search Results window. Select one or more users and/or groups as follows:

 NOTE: A single Grant Access to Users/Groups assertion will be added to the security policy for each user and/or group selected in the Search Identity Provider form. A security policy with more than one Grant Access to Users/Groups assertion must organize the identities into an "At least one assertion must evaluate to true" assertion folder with its required authentication method assertion. To add a folder to the Policy Development window, right-click any assertion and select "Add 'One or More' Assertion" from the drop-down menu. Drag and drop a user or group into the folder as required. For more information, see Security Policy Organization and Rules and Configure and Edit Security Policies.

Select a Single User or Group

To select a single user or group for the security policy, click the user or group's name and click [Select]. The Search Identity Provider form closes and an assertion for the user or group is added to the Policy Development window.

Select Multiple Users and/or Groups

    1. Select multiple users or groups for the security policy by one of two methods:

      • To select a consecutive range of users or groups, hold down your left mouse button as you move from the first user or group to the last user or group in your target range. Or, press [Shift] on your keyboard, left-click the first user or group in the range, and then left-click the last user or group in the range. Click [Select]

      • To add users or groups placed throughout the list, press [Ctrl] on your keyboard as you click each user or group name that you want to add. Click [Select].

    1. The Search Identity Provider form closes and an assertion for each user or group is added to the Policy Development window. Repeat the process as required.

 

 NOTE: You cannot edit the Grant Access to Users/Groups assertion(s). To remove the assertion, right-click the assertion name in the Policy Development window. A drop-down menu appears. Select [Delete Assertion]. To view policy assertions by the configured users and/or groups in the policy (read-only), click the [Identity View] button on the Policy Tool Bar. To switch back to the Policy Development window, click the [Policy View] button.

HTTP Basic

 NOTE: Since the HTTP Basic assertion requires the insertion of the password into the service message, it is recommended that you also select the SSL Transport assertion when using basic HTTP authentication. See Configure and Edit Transport Layer Security (TLS) Assertions for information about the SSL Transport assertion.

The HTTP Basic assertion allows you to require basic HTTP authentication — user name, plain text password, and the authentication realm — as a string in the Web service or XML application request headers. This assertion is a credential source that saves the user name and password from the HTTP headers for later authentication and authorization via the Grant Access to Users/Groups assertion. For information on the Grant Access to Users/Groups assertion, see Grant Access to Users/Groups.

 

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

 

 NOTE: You cannot edit the HTTP Basic assertion. To remove the assertion, right-click the assertion name in the Policy Development window. A drop-down menu appears. Select [Delete Assertion].

HTTP Digest

 NOTE: You cannot use the HTTP Digest assertion with the SOAP Request with Attachment assertion. For more information about the SOAP Request with Attachment assertion, see Configure and Edit Message Validation/Transformation Assertions.

The HTTP Digest assertion allows you to require digest HTTP authentication — user name, password, and the authentication realm — in the request headers of a "SHA" digest form for the Web service or XML application. This assertion is a credential source that saves the user name and digested password from the HTTP headers for later authentication and authorization via the Grant Access to Users/Groups assertion. For information on the Grant Access to Users/Groups assertion, see Grant Access to Users/Groups.

 

 NOTE: You cannot use the HTTP Digest assertion with External Identity Provider (EIP) users or groups. For more information about EIPs, see Add and Edit External Identity Providers.

 

  • Add the assertion to the Policy Development window by one of two methods:

  • The assertion is added to the Policy Development window.

 

 NOTE: You cannot edit the HTTP Digest assertion. To remove the assertion, right-click the assertion name in the Policy Development window. A drop-down menu appears. Select [Delete Assertion].

HTTP Client Certificate

This assertion is a credential source that saves the client certificate from the SSL-TLS handshake for later authentication and authorization via the Grant Access to Users/Groups assertion. WSS Signature

 NOTE: The "WSS" in the WSS Signature assertion stands for "WS-Security." The assertion supports version 1.0 of the WS-Security standard. The SecureSpan™ Solution creates and uses X.509 v3 certificates.

The WSS Signature assertion allows you to require that the Web service or XML application request message:

  • Include an X.509 BinarySecurityToken containing a client certificate

  • Have at least one element signed by that client certificate's private key as a proof of possession of the private key for the client certificate.

The WSS Signature assertion is a credential source that saves the certificate from the X.509 BinarySecurityToken for later authorization via the Grant Access to Users/Groups assertion. For information on the Grant Access to Users/Groups assertion, see Grant Access to Users/Groups.

 

The WSS Signature assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab. See Configure and Edit XML Security Assertions for more information.

 

 NOTE: The WSS Signature assertion, by itself, does not require that the request message contain a timestamp, and does not check the validity of any timestamp that might be present. To protect against stale or replayed messages, use the WSS Signature assertion with the WSS Replay Protection assertion. See Configure and Edit XML Security Assertions for more information about the WSS Replay Protection assertion.

 

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

 

 NOTE: You cannot edit the WSS Signature assertion. To remove the assertion, right-click the assertion name in the Policy Development window. A drop-down menu appears. Select [Delete Assertion].

WS Secure Conversation

 NOTE: The WS Secure Conversation assertion supports the WS-Security 1.0 standard.

The WS Secure Conversation assertion allows you to require that request and response messages be secured using a secure conversation as opposed to session-less security. Specifically, messages must:

  • Include a "SecurityContextToken" referencing an already-established WS-SecureConversation session

  • Include at least one element signed with the shared secret from this session as proof of possession of the session shared secret

  • Be delivered to the same SecureSpan™ Gateway node on which the session is established. In practice, this typically requires that the request message arrive via HTTP(S) routing rather than JMS routing. For more information about the message routing assertions, see Configure and Edit Message Routing Assertions.

The WS Secure Conversation assertion is a credential source that saves the user that owns the session for later authorization via the Grant Access to Users/Groups assertion. For information on the Grant Access to Users/Groups assertion, see Grant Access to Users/Groups. The WS Secure Conversation assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab. See Configure and Edit XML Security Assertions for more information.

 

 NOTE: The WS Secure Conversation assertion, by itself, does not require that the request message contain a timestamp, and does not check the validity of any timestamp that might be present. To protect against stale or replayed messages, use the WS Secure Conversation assertion with the WSS Replay Protection assertion. See Configure and Edit XML Security Assertions for more information about the WSS Replay Protection assertion.

 

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

 

 NOTE: You cannot edit the WS Secure Conversation assertion. To remove the assertion, right-click the assertion name in the Policy Development window. A drop-down menu appears. Select [Delete Assertion].

SAML Security

 NOTE: The SAML Security assertion supports the SAML 1.1 standard. To avoid constraint conflicts, only a single SAML Security assertion should be present in a security policy.

The SAML Security assertion allows you to require single sign-on authentication and authorization for a Web service or XML application. Using the required constraints, SAML (Security Assertions Markup Language) validates a ticket to ensure that it falls within the approved time and date period. If validation succeeds, then the SecureSpan™ Gateway passes the message through to the service. If validation fails, the Gateway returns a SOAP fault.

 

 NOTE: The acceptable time and date parameters are specified in the "Not Before" and "NotOnOrAfter" attributes of the "saml:Conditions" in the SOAP header of the XML document.

 

The SAML Security assertion is a credential source that saves the subject certificate and issuer information for later authorization via the Grant Access to Users/Groups assertion. For information on the Grant Access to Users/Groups assertion, see Grant Access to Users/Groups. The SAML Security assertion can be used in tandem with the WSS Replay Protection, Sign Request Element, and Encrypt Response Element assertions from the XML Security category of the [Assertions] tab. See Configure and Edit XML Security Assertions for more information.

 

  • Add the assertion to the Policy Development window by one of two methods:

    • Drag and drop the assertion from the [Assertions] tab into the Policy Development window, or  

    • Highlight the assertion in the [Assertions] tab and click the  (Add Assertion) button in the Assertion Tool Bar.

  • The assertion is added to the Policy Development window.

 

 NOTE: You cannot edit the SAML Security assertion. To remove the assertion, right-click the assertion name in the Policy Development window. A drop-down menu appears. Select [Delete Assertion].
