About Stored Policies

NOTE: Policies are initially downloaded and saved in the SecureSpan™ Bridge based on the Policy Attachment Key in the initial service request message. A Policy Attachment Key is the combination of the Body Namespace value, SOAPAction header, and Proxy URI in the service request. The security requirements for a service are defined in the SecureSpan™ Manager and saved in the applicable SecureSpan™ Gateway or manually imported from another location. See the SecureSpan Manager documentation for more information.

Message security requirements are automatically downloaded from the SecureSpan™ Gateway during the initial consumption of a service, or manually imported from another location and stored as a partial policy in the SecureSpan™ Bridge (see Import/Export Policies). When downloaded from the SecureSpan Gateway, the stored policy will contain the security requirements applicable to the user account that was used to download the policy. When downloaded or imported from any location as a formatted XML file, the stored policy will only contain the assertions applicable to the type of message decoration that will be performed by the SecureSpan Bridge.

 

Policies are viewed by Policy Attachment Key in the Services with Cached Policies table and Associated Policy window in the [Service Policies] tab in the Gateway Account Properties form.

 

The [Service Policies] Tab in the Gateway Account Properties Form

 

About the Services with Cached Policies Table

When the SecureSpan™ Bridge first connects to a service protected by the SecureSpan™ Gateway in the Gateway Account, the policy in force for the service will be automatically downloaded to the Services with Cached Policies table:

  • The Body Namespace column lists the service namespace URI of the payload element of a SOAP message. The payload element is the first child of the SOAP Body element in a service message, such as “http://warehouse.acme.com/ws.” For non-SOAP or XML application messages, the body namespace is always an empty string

  • The SOAPAction column contains the content of the SOAPAction header that accompanies the service message. For example, “http://warehouse.acme.com/ws/currentOrders”

NOTE: Only the value in the SOAPAction field must be surrounded by quotes (“”). When editing or configuring a Policy Attachment Key, do not include quotes around the values in the Body Namespace field or Proxy URI field. See Manage Stored Policies for more information.

  • The Proxy URI column contains the extra information in the path of the request into the SecureSpan Bridge. For example, if a request is made to the SecureSpan Gateway identified by the proxy URL “http://localhost:7700/ssg/soap,” then the Proxy URI for the request would be the string “/ssg/soap”

  • The Match Type column contains a drop-down list of two Policy Attachment Key match options. Select the “equals” option to require that request messages exactly match the Policy Attachment Key for the service. Select the “begins with” option to allow all request messages that match the start of the corresponding Policy Attachment Key settings for the service. For more information, see Manage Stored Policies

  • The Lock column contains a check box for each row corresponding to a service with a cached policy. When checked, the policy will automatically be saved to disk between SecureSpan Bridge invocations and cannot be overridden by the SecureSpan Gateway. For more information, see Manage Stored Policies.

 

The Policy Attachment Key for a policy is the combination of the Body Namespace value, SOAPAction header, and Proxy URI. Depending on the setting in the Match Type column, a service request message must either exactly match the entire or beginning values of the Policy Attachment Key in order to be processed under the appropriate policy. The values that make up the Policy Attachment Key for a service can be edited. See Manage Stored Policies for more information.

 

Select a row to view the read-only assertion details for the downloaded or imported policy in the Associated Policy window. See About the Associated Policy Window below for more information. For information on configuring, importing, exporting, and deleting the policies in the Services with Cached Policies table, see Manage Stored Policies and Import/Export Policies.

 

About the Associated Policy Window

NOTE: For information on configuring, importing, exporting, and deleting policies in the SecureSpan™ Bridge, see Manage Stored Policies and Import/Export Policies. For information about the assertions that make up a policy, see the SecureSpan Manager documentation.

The Associated Policy window displays the read-only assertions that make up the policy for the service selected in the Services with Cached Policies table. The assertions are presented in a tree structure similar to the assertion tree in the SecureSpan™ Manager’s Policy Development window.

 

When a policy is stored in the SecureSpan™ Bridge, only the security requirements applicable to the type of message decoration that will be performed by the SecureSpan Bridge are referenced in the Associated Policy window. Other assertions will be identified as unsupported assertions and ignored by the SecureSpan Bridge during the message decoration process. The types of policy assertions that are viewable in the Associated Policy window are:

  • SSL Requirement: If SSL is required by the service, then the SecureSpan Bridge will negotiate a certificate-based security session with the SecureSpan™ Gateway. See About SSL/TLS Security Sessions for more information

  • Authentication Method: The SecureSpan Bridge will send the credentials for connecting to a service based on the method of authentication specified in the policy. If a client certificate-based authentication method is required by the policy, then the SecureSpan Bridge will negotiate a certificate-based security session with the SecureSpan Gateway. If the SecureSpan Bridge must establish a WSS Secure Conversation security session with the SecureSpan Gateway, then the SecureSpan Bridge will contact the SecureSpan Gateways WS-Trust security token service to establish a WS-SecureConversation SecurityContext. The SecurityContext will be used for sending service requests to the target SecureSpan Gateway. If XPath evaluation is required, then the SecureSpan Bridge will verify that the request or response message contains a WS-Security BinarySecurityToken containing an X.509 certificate, and that the private key for this certificate was used to sign at least one element of the message. When the SecureSpan Bridge generates a message according to a policy with this requirement, it will always sign the timestamp and any other elements required by the policy

NOTE: The SecureSpan™ Bridge is only aware of the credential format requirements for a service, not the actual credentials (user names and passwords) that are required to connect to the service. The SecureSpan Bridge only transmits the credentials stored in the target Gateway Account.

  • XML Signature or Encryption Requirement: If required, the SecureSpan Bridge will decorate a request message (or elements of a message) according to the request signature or encryption requirements of a policy. If response message encryption or signing is required, then the SecureSpan Bridge will undecorate the response message from the SecureSpan Gateway accordingly

  • Replay Protection Requirement: If replay protection is required by a policy, then the SecureSpan Bridge will include a signed, high-resolution timestamp in the request message. The SecureSpan Bridge will also confirm the timestamp and message ID of the response message before forwarding it to the client application. The timestamps guard against the replay of old request and response messages.

 

For information about the assertions that make up a policy, see the SecureSpan™ Manager documentation.
