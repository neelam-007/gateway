Create Trusted Gateway Accounts

TIP: All Gateway Accounts will normally be Trusted Gateway Accounts unless the SecureSpan™ Bridge is being used in an identity bridging configuration (see Identity Bridging).

In the SecureSpan™ Bridge, a “Trusted Gateway” is a SecureSpan™ Gateway on which you have an established identity, such as a user or group account in the Internal Identity Provider, in an LDAP Identity Provider, or in a non-LDAP identity provider such as Netegrity SiteMinder™. Only a Trusted Gateway is allowed to know the password configured for its corresponding Gateway Account in the SecureSpan Bridge. A Trusted Gateway can automatically establish the authenticity of its server certificate, and can then authenticate requests using the established user name and password, if provided, to issue a client certificate or to provide access to services.  

 

FYI: For information about identity providers and the optional add-on assertions that permit connections to external programs, or non-LDAP identity providers, see the SecureSpan™ Manager documentation.

 

As part of an identity bridging-configuration, a Trusted Gateway can act as a SAML security token service, allowing access to Federated Gateways on which no account exists but which trust security tokens issued by the Trusted Gateway. For information about the role of Gateway Accounts in an identity bridging-configuration, see Identity Bridging.

 

A new Trusted Gateway Account is configured with the SecureSpan Bridge’s Gateway Account Properties form. Accessed from the Create Gateway Account form when configuring a new Gateway Account, the Gateway Account Properties form configures the required and optional settings for the proxy relationship. Use the SecureSpan Bridge’s Create Gateway Account form to establish a new Trusted Gateway Account for a SecureSpan Gateway as follows:

  1. Open the Create Gateway Account form by one of two methods:

    • Click the [New] button on the Main Tool Bar or,

    • Click [File] > New from the Main Menu.

  1. The Create Gateway Account form appears. Select the Trusted Gateway option.

  2. Click [Create]. The Gateway Account Properties form appears.

  3. In the [General] tab of the Gateway Account Properties form, enter the URL of the SecureSpan Gateway into the Gateway Host Name field in the following format: machinename.domain.com.

NOTE: The host name must exactly match the name of the SecureSpan™ Gateway machine assigned during the creation of the SecureSpan Gateway server certificate. To check the host name in the server certificate after it has been discovered, click the [View Server Certificate] button. Verify that the SecureSpan Gateway DNS information or SecureSpan™ Bridge hosts file information for connecting to the SecureSpan Gateway server is correct.

  1. Click the [Identity] tab. Do one of the following:

NOTE: If a client certificate has been issued for a set of credentials in the SecureSpan™ Bridge, then the user name and password values cannot be edited. In this case, a new relationship must be created to change the credentials. For information on importing a client certificate, see Require Credentials from the Client Application. When credentials are not associated with a client certificate, then the user name and password can be changed as outlined in Manage Existing Gateway Accounts.

Enter Credentials

Enter a user name and password for the Gateway Account into the respective User Name and Password fields. The credentials are stored in the SecureSpan Gateway’s Internal Identity Provider, in an LDAP Identity Provider, or in a non-LDAP identity provider, such as Netegrity SiteMinder™, that the SecureSpan Gateway authenticates against. Proceed to Step #6.

NOTE: You can check the Save Password check box to save the password in a SecureSpan™ Bridge configuration file. When checked, the SecureSpan Bridge will not prompt for a password during subsequent launches. However, configuring the SecureSpan Bridge to save the password associated with a Gateway Account has security implications. The saved password is not encrypted, but saved in obfuscated form in the ssg.xml file in the local .l7tech directory. An attacker may be able to gain access to this file and read the password. If security is a concern for your organization, then it is strongly recommended that you do not check the Save Password check box.

Enter a User Name Only

Enter a user name in the User Name field while leaving the Password field blank (empty). The SecureSpan™ Gateway will prompt for a password during the initial consumption of a service. The password entered at that time will be stored by the SecureSpan Bridge for the remainder of the session. Proceed to Step #6.

Do Not Enter Credentials

Leave the User Name and Password fields blank (empty) to not require credentials in the Gateway Account. When credentials are not specified in the SecureSpan™ Bridge, the SecureSpan Bridge will still prompt for a user name and password in order to attempt automatic server certificate discovery when consuming a service for the first time, even when the policy for that service (as constructed in the SecureSpan™ Manager) does not require authentication. Proceed to Step #6.

TIP: If you do not have credentials, or if credentials are not required by the service, then do one of the following to by-pass the credential prompt and/or server certificate discovery process: 1) When the credential form appears, leave the User Name and Password field blank. Click [OK]. Manually inspect the server certificate when prompted; and/or 2) In the [Bridge Policy] tab for the Gateway Account, uncheck the Use SSL By Default checkbox to avoid using SSL for the initial request, thereby deferring the need for server certificate discovery.

Require Credentials from the Client Application

NOTE: Use this credential method when, for confidentiality reasons, you do not want to store credentials on the SecureSpan™ Bridge. In this credential method, the SecureSpan™ Gateway must be able to authenticate the credentials against its Internal Identity Provider or an External LDAP Identity Provider. For information about identity providers, see the SecureSpan™ Manager documentation.

In this credential method, the client application will send credentials with the service request to the SecureSpan™ Bridge using the HTTP Basic format. The SecureSpan Bridge will forward the credentials and request to the SecureSpan™ Gateway for authentication. To use credentials from the client application that sends service requests to the SecureSpan Bridge, do the following:

    1. Leave the User Name and Password fields blank (empty).

    2. Do not check the Save Password check box.

    3. Check the Use Credentials from Client (HTTP Basic Authentication) check box.

NOTE: Steps #4-10 for importing a client certificate below can be omitted if no client certificate-based features will be needed in the Gateway Account. For example, if the target service uses password-based authorization over SSL, then no client certificate would be necessary in the Gateway Account. If applicable, proceed to Step #6.

    1. Click [View Client Certificate]. The Client Certificate Not Found form appears. Do one of the following:

Import the Client Certificate

Import a PKCS #12 keystore certificate for the Gateway Account:

      1. Click [Import Client Certificate]. The Select Client Certificate form appears.

NOTE: You must have access to a pre-existing private key PKCS#12 keystore certificate (such as "cert.p12"). The certificate can be self-signed or signed by a trusted certificate authority. You must also have the PKCS#12 file password.

      1. Navigate to the location of the certificate. Select the file. Click [Import Certificate].

      2. The Enter Pass Phrase for this PKCS#12 File form appears. Enter the PKCS#12 file password into the Password field. Click [OK].

      3. The Enter New Password for Gateway Localhost form appears. Assign a password to the Gateway Account that will be used to encrypt the newly-imported private key. Enter the password into the Password field. Repeat the password in the Verify Password field.

      4. Click [OK]. A pop-up message confirms the successful import of the certificate or displays import errors.

      5. Click [OK] to close the message. Repeat the import process to correct any errors. If necessary, contact a Technical Support Representative as outlined in Technical Support. When the import has been confirmed, proceed to Step #5 below.

Request Client Certificate from the SecureSpan Gateway

Download the PKCS # 12 keystore certificate from the SecureSpan Gateway:

      1. Click [Request Client Certificate from Gateway]. The Log On to Gateway form appears.

      2. Enter the SecureSpan Gateway password into the Password field.

      3. Click [OK].

    1. The User Name and Password fields are automatically populated with the user name and password specified in the imported PKCS#12 keystore. Proceed to Step #6 below.

NOTE: Credentials provided by a client application cannot be changed in the SecureSpan™ Bridge. If credential requirements do change, then you must remove and re-create the Gateway Account to reflect the new credentials.

 

  1. Click the [Network] tab. In the Outgoing Requests to the Gateway section:

    • Optionally change the default port settings (HTTP: 8080 and HTTPS: 8443; the Gateway Uses Standards Ports option is selected by default) of the SecureSpan Gateway to custom port numbers as required by your network. To change the default ports:

      1. Select the Gateway Requires Custom Ports option.

      2. Enter the applicable port numbers into the HTTP Port and HTTPS Port fields.

    • Optionally specify one or more SecureSpan Gateway IPs (the Look Up IP Addresses in DNS option is selected by default) that the SecureSpan Bridge can use for routing service messages and accepting policies and required certificates:

TIP: Multiple hard-coded IPs allow the SecureSpan™ Bridge to bypass the DNS when a host name does not resolve to an appropriate SecureSpan™ Gateway IP address. Manual IP configuration can also be used for failover protection, to connect to a SecureSpan Gateway whose certificate does not contain a domain name, or to connect to a SecureSpan Gateway whose certificate contains an incorrect domain name. In most cases, however, it is best to get IP information from the DNS.

      1. Select the Use the Following IP Addresses option.

      2. Click [Add]. The Add IP Address form appears.

      3. Enter the SecureSpan Gateway IP address into the IP Address field. Click [OK].

      4. The IP address appears in the field. If required, repeat Steps #2-3 to add another SecureSpan Gateway IP address.

      5. When all IP addresses have been added, select one of the following processing options from the Failover Strategy drop-down list:

        • When the Ordered Sticky with Failover processing option is selected, the SecureSpan Bridge will send service messages to the first IP in the list, and continue to do so until the IP does not respond. The SecureSpan Bridge will then proceed to the next IP in the list

        • When the Random Sticky with Failover processing option is selected, the SecureSpan Bridge will choose an IP randomly at the beginning of each session and use it for the duration of the session. If the chosen IP address fails, then the SecureSpan Bridge will select another IP at random

        • When the Pure Round Robin processing option is selected, the SecureSpan Bridge will rotate through the IP list on a request-by-request basis (round-robin) from the first IP, to the second IP, and so on. When the end of the IP list is reached, the SecureSpan Bridge will cycle back to the top and start again.

  1. Click the [Bridge Policy] tab. By default, the Use SSL by Default check box is checked. Optionally uncheck the check box to turn off the SSL download of policies to the SecureSpan Bridge.

NOTE: If you uncheck the Use SSL by Default check box, then the initial service request will not be secured. If the target service is secure, then you may be prompted for a password to download the policy for the target service, but you may have unwittingly transmitted unsecured sensitive data as part of the initial request.

  1. Click the [Service Policies] tab. The Services with Cached Policies table is the repository for the policies, listed by Policy Attachment Key, being applied to service messages by the SecureSpan Bridge. The Associated Policy window contains assertion-level information about a policy selected in the Services with Cached Policies table. For more information, see About Stored Policies. For information on configuring, editing, and deleting policies, see Manage Stored Policies. For information on importing or exporting policies, see Import/Export Policies.

NOTE: If automatic policy discovery will be used in the Gateway Account, then no additional configuration in the [Service Policies] tab is required. When automated, the SecureSpan™ Gateway will automatically populate the [Services Policies] tab with any policies required by its services.

  1. Click [OK]. The SecureSpan Bridge automatically adds the new Gateway Account to the Account List with a unique identifier. The format of the identifier in the Proxy column of the Account List will be “gateway<sequential number>.” The “sequential number” part of the identifier is assigned based on the number of Gateway Accounts configured in the SecureSpan Bridge. The first configured Gateway Account, for example, is assigned a unique identifier of “gateway1” and considered the default Gateway Account. Subsequent Gateway Accounts will contain a number reflecting their configuration order and will not be assigned a default status. To change the default Gateway Account in the SecureSpan Bridge, or to remove the default status from the Account List, see Manage Existing Gateway Accounts.
