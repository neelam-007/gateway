SecureSpan™ Gateway Accounts Overview

 NOTE: The term "identity" includes both users and groups, the term "user" can represent an individual human or machine, the term “service” includes both Web services and XML applications, and the term "SecureSpan™ Gateway" includes all of the SecureSpan Gateway products.

The SecureSpan™ Bridge acts as a service proxy by establishing a PKI-based trust relationship with one or more SecureSpan™ Gateways. Known as a “Gateway Account,” the proxy relationship in the SecureSpan Bridge is essentially a setting that combines a SecureSpan Gateway with a single set of credentials that are used to process service requests.

 

A single SecureSpan Gateway can be accessed by multiple SecureSpan Bridges. Furthermore, a single SecureSpan Bridge can access more than one Gateway Account — each with different credentials — for a single SecureSpan Gateway. In a configuration where the SecureSpan Bridge is servicing requests from a JSP page hosted on IIS, for example, the JSP page could be programmed to send requests in three ways. Requests from a certain IP address could be sent as “guest” users, requests with a user name and password could be sent as “member” users, and requests from inside the corporate firewall could be sent as “employee” users and routed to a separate internal SecureSpan Gateway. In this scenario, each way of sending requests requires a different set of credentials and a separate Gateway Account in the SecureSpan Bridge. For information about the role of Gateway Accounts in an identity bridging-configuration, see Gateway Accounts and Identity Bridging.

 

Two different types of Gateway Accounts are available in the SecureSpan Bridge:

Trusted Gateway Account

 TIP: All Gateway Accounts will normally be Trusted Gateway Accounts unless the SecureSpan™ Bridge is being used in an identity bridging-configuration.

In the SecureSpan™ Bridge, a “Trusted Gateway” is a SecureSpan™ Gateway on which you have an established identity, such as a user or group account in the Internal Identity Provider, in an LDAP Identity Provider, or in a non-LDAP identity provider such as Netegrity SiteMinder™. Only a Trusted Gateway is allowed to know the password configured for its corresponding Gateway Account in the SecureSpan Bridge. A Trusted Gateway can automatically establish the authenticity of its server certificate, and can then authenticate requests using the established user name and password, if provided, to issue a client certificate or to provide access to services.  

 

 NOTE: For information about identity providers and the optional add-on assertions that permit connections to external programs, or non-LDAP identity providers, see the “SecureSpan™ Manager User Manual” or program-based “SecureSpan™ Manager Help System.”

 

As part of an identity bridging-configuration, a Trusted Gateway can act as a SAML security token service, allowing access to Federated Gateways on which no account exists but which trust security tokens issued by the Trusted Gateway. For more information, see Gateway Accounts and Identity Bridging and Identity Bridging Overview. For information on configuring a Trusted Gateway, see Create Trusted Gateway Accounts.

Federated Gateway Account

 TIP: A Federated Gateway Account will typically only be used in an identity bridging-configuration. For more information, see Gateway Accounts and Identity Bridging and Identity Bridging Overview.

In the SecureSpan™ Bridge, a “Federated Gateway” is a SecureSpan Gateway on which you do not have an established identity in its Internal Identity Provider, in an LDAP Identity Provider, or in a non-LDAP identity provider. A Federated Gateway authenticates requests by accepting SAML security tokens issued by a security token service on which you do have an established identity, or user or group account. The security token service can be either a third-party WS-Trust server or an already-configured Trusted Gateway. For information on configuring a Federated Gateway, see Create Federated Gateway Accounts.

 

See Manage Gateway Accounts for information on changing a Gateway Account password, editing and deleting a Gateway Account, or assigning and deleting the default status applied to a Gateway Account.
