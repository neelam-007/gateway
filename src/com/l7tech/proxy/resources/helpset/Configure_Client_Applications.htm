Configure Client Applications

 NOTE: For detailed SecureSpan™ Bridge configuration information and instructions, see the “SecureSpan™ Bridge and SecureSpan™ Manager Installation Manual.”

The SecureSpan™ Bridge acts as a proxy between a client application and a SecureSpan™ Gateway-protected service. The client application that communicates with the SecureSpan Bridge could be anything from a small application that sends service requests to a large-scale Web server that re-directs service requests that originate from a Web page. Regardless of the source of service requests, referred to as the “client application” for brevity, the client-side configuration of the system must abide by the following to facilitate communication with the SecureSpan Bridge:

Physical Location

 NOTE: When installed on a multi-user machine, the SecureSpan™ Bridge may be accessible by unauthorized local users.

The client application and SecureSpan™ Bridge must reside on the same physical machine. To prevent hijack and the receipt of requests from rogue machines, the SecureSpan Bridge will only accept requests from the special address “localhost” with the IP address “127.0.0.1.” This address should only be accessible by other programs running on the same machine as the SecureSpan Bridge.

 

URL and Port Designation

 NOTE: WARNING: If the SecureSpan™ Bridge is left running on a Windows XP or Linux system, another user can logon and use the current SecureSpan Bridge credentials to access services using the first user’s credentials. In other words, once port 7700, or an alternate port, is open to receive traffic, any user can logon and send a request to that port for processing. It is strongly recommended, therefore, that users shut down the SecureSpan Bridge when leaving their desks. If the SecureSpan Bridge was configured to run on a different port than the default 7700 port number, then substitute the alternate port number into the following localhost URLs.

The client application must be configured to send service requests to the SecureSpan™ Bridge localhost port. By default, the SecureSpan Bridge is configured to run and listen for traffic on port “7700.” Therefore, the client application must be configured to send service requests to one of the following localhost URLs: “http://localhost:7700” or “http://127.0.0.1:7700.”

 

If the SecureSpan Bridge contains more than one Gateway Account, then the localhost URL can be narrowed to specify that the service request be sent to a specific SecureSpan™ Gateway, such as “http://localhost:7700/gateway<sequential number>.” The “gateway<sequential number>” portion of the URL is assigned based on the number of Gateway Accounts configured in the SecureSpan Bridge. See Create Trusted Gateway Accounts or Create Federated Gateway Accounts for more information.

 

Message Formats

The SecureSpan™ Bridge accepts HTTP POST request messages from the client application. The format of a typical SOAP HTTP POST request message may be:

 

--------------------------------------------------------------------------------------

POST /gateway3 HTTP/1.0

SOAPAction: “http://example.com/services/ACME#listProducts”

Content-Type: text/xml

 

<?xml version="1.0" encoding="UTF-8"?>

<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">

    <soap:Body>

        <listProducts xmlns="http://warehouse.acme.com/ws"/>

    </soap:Body>

</soap:Envelope>

-----------------------------------------------------------------------------------------

 

 NOTE: The message format example is a SOAP request message designated for a SOAP Web service. XML application or non-SOAP service request messages will probably not contain a SOAPAction header or SOAP elements like “Envelope” and “Body.” For more information, see About XML Application or Non-SOAP Messages below.

 

The SecureSpan Bridge determines the Gateway Account for the target SecureSpan™ Gateway, “gateway3” in the example, from the “POST /gateway3 HTTP/1.0” header in the request message, yielding one of two outcomes:

  1. If a Gateway Account match for “gateway3” is found, then the SecureSpan Bridge will analyze and match the request message to a Policy Attachment Key.

  2. If a Gateway Account match for “gateway3” is not found, then the SecureSpan Bridge will process the request using the default Gateway Account. If a default Gateway Account does not exist, then an error message will appear and the request will fail.

 

When a Gateway Account match is found, the SecureSpan Bridge will analyze the request message to determine the policy under which the message must be processed. Three values in the request message are analyzed to determine the Policy Attachment Key that identifies the target service policy:

  • The Proxy URI, or “/gateway3” in the example message, identifies the target SecureSpan Gateway

  • The SOAPAction, or “http://example.com/services/ACME#listProducts” in the example message, is the URL of the target service

 NOTE: The SOAPAction header will be blank if the target service in an XML application, also known as a non-SOAP service.

  • The Body Namespace, or “http://warehouse.acme.com/ws” in the example message, is the payload element's namespace URI.

 

When the Policy Attachment Key in the request message is matched to a Policy Attachment Key in the Gateway Account, the SecureSpan Bridge will decorate the request message according to the security requirements in the associated policy. For more information about message routing, see About Message Routing. For more information about Policy Attachment Keys and the message decoration process, see About Stored Policies and Manage Stored Policies.

About XML Application or Non-SOAP Messages

The format and analysis of an XML application or non-SOAP message differs from a SOAP message, or a message designated for a Web service. By design, a non-SOAP message will not contain a SOAPAction value or, since the payload element is found under the SOAP body element, a Body Namespace value. When assessing a non-SOAP message, the SecureSpan™ Bridge considers the non-existent SOAPAction value and Body Namespace value as empty strings in the Policy Attachment Key. Consequently, the SecureSpan Bridge will match the Proxy URI in the request message to the Policy Attachment Keys in the Gateway Account. Once a match is found, the SecureSpan Bridge will decorate the request message according to the security requirements in the associated policy.
