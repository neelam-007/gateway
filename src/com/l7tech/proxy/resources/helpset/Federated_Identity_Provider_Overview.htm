Federated Identity Provider Overview

 NOTE: The generic "Trusted Gateway A" and "Federated Gateway B" references are used in the identity bridging examples, workflows, and instructions. The Trusted Gateway A is the certificate authority (CA) that issues and manages security credentials and that is responsible for authentication. The Federated Gateway B is the Web service provider that is responsible for authorization.

In an identity bridging configuration, the Federated Identity Provider (FIP) allows the Federated Gateway B (authorization domain) to authorize requests containing credentials originating in the Trusted Gateway A (authentication domain). Credentials may be X.509 certificates signed by trusted certificate authorities (CAs) (see Identity Bridging Workflow Using an X.509 Certificate) or SAML tokens signed by a SecureSpan™ Gateway or other LDAP Identity Provider (see Identity Bridging Workflow Using SAML). Alternatively, a Federated Identity Provider may not contain any certificates.

 

The trust store in the Federated Gateway B is the repository for the certificates from other security domains that may be required by the FIP in an identity bridging configuration. Certificates are defined and added to the trust store with the Add Certificate Wizard prior to creating the FIP in the Federated Gateway B (see Certificates Overview). The chosen credential source and optional configuration elements outlined in Identity Bridging Workflow Using an X.509 Certificate or Identity Bridging Workflow Using SAML determine certificate and FIP configuration details. Once the trusted certificates are added to a new FIP, federated users, groups, and/or virtual groups can be created to authorize corresponding users, groups, or credential patterns in the Federated Gateway B security domain. See Add and Edit Federated Identity Provider Users and Groups for more information.
