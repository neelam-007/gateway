About Request Message Routing

NOTE: The SecureSpan™ Bridge can accept decorated or undecorated XML or SOAP messages from client applications. See Configure Client Applications for more information. For information about SecureSpan Bridge policies, see About Stored Policies.

When the SecureSpan™ Bridge receives a service request message from a client application, it decorates the message according to the security requirements in its saved policy. It then routes the decorated request message to the SecureSpan™ Gateway. The SecureSpan Gateway analyzes the decoration to confirm that it conforms to the policy for the target service, yielding one of three outcomes:

 

Request Message Conforms to the Policy

If the decorated request message conforms to the policy requirements stored in the target SecureSpan™ Gateway, then the request message will be processed. Once completed, the SecureSpan Gateway will send a service response message to the SecureSpan™ Bridge. The SecureSpan Bridge will un-decorate the response message as required before routing it to the client application.

Request Message Does Not Conform to the Policy

If the decorated request message does not conform to the policy requirements stored in the target SecureSpan™ Gateway, then the request message will be rejected by the SecureSpan Gateway. In this case, the SecureSpan Gateway will note the out-of-date policy and will automatically send the updated policy to the SecureSpan™ Bridge. The SecureSpan Bridge then re-decorates the request message according to the security requirements of the new policy before re-routing it to the SecureSpan Gateway.

 

NOTE: If the target policy in the SecureSpan™ Bridge is locked, then the SecureSpan™ Gateway will not be able to automatically update the outdated policy. In this case, the service request will fail. For information about locked and unlocked policies, see Manage Stored Policies.

 

Request Message Conforms to the Policy, but is Invalid

If the decorated request message conforms to the policy requirements stored in the target SecureSpan™ Gateway, but contains an invalid decoration, user name and/or password, or other invalid component, then the SecureSpan Gateway will reject the message. Confirm the non-policy details required by the SecureSpan Gateway and update the Gateway Account properties accordingly. If errors still occur, contact a Technical Support Representative as outlined in Technical Support.

 

About Response Message Routing

NOTE: The SecureSpan™ Bridge sends the response message to the client application as an un-decorated plain text XML message without a SOAP security header.

Upon receiving a service response message from the SecureSpan™ Gateway, the SecureSpan™ Bridge un-decorates the message, removing any security requirements applied to conform to the service’s policy. The SecureSpan Bridge will then send the service response message to the client application, concluding the service transaction process.
