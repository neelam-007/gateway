Configure Gateway Accounts for Identity Bridging

NOTE: The generic "Trusted Gateway A" and "Federated Gateway B" references are used in the identity bridging examples, workflows, and instructions. The Trusted Gateway A is the certificate authority (CA) that issues and manages security credentials and that is responsible for authentication. The Federated Gateway B is the Web service provider that is responsible for authorization.

In an identity bridging configuration, the SecureSpan™ Bridge delegates authentication to the Trusted Gateway A while preserving authorization for the Federated Gateway B hosting the Web service.

 

The SecureSpan Bridge interfaces to the requestor’s SAML or X.509 certificate authorization source that validates an authentication. Once the SecureSpan Bridge receives the authentication assertion or token, it will automatically embed this evidence and then sign the SOAP message. Since the Federated Gateway B will have already established a certificate-based trust relationship with the Trusted Gateway A's authentication source, the Federated Gateway B can then use this evidence to securely authorize access to the protected Web service. See the SAML credential source and X.509 certificate credential source diagrams in Identity Bridging for more information about the role of the SecureSpan Bridge in an identity bridging-configuration.

 

As part of the SAML and X.509 credential source workflows, Gateway Accounts corresponding to the Trusted Gateway A and/or Federated Gateway B must be configured in the SecureSpan Bridge. Once the required Gateway Accounts are configured, a Web service can be consumed as part of the identity bridging arrangement. See Consume Shared Web Services for more information.

 

For information about identity bridging using SAML or an X.509 certificate, see Workflow Using SAML or Workflow Using an X.509 Certificate, respectively.
