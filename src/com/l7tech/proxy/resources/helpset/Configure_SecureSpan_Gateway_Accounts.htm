Configure Gateway Accounts

There are two ways to configure the SecureSpan™ Bridge to participate in an identity bridging arrangement:

  • If the SecureSpan Bridge has access to two SecureSpan™ Gateways — one as the Trusted Gateway A in the authentication domain and the other as the Federated Gateway B in the authorization domain — then configure the Bridge using the Full Identity Bridging method below. This method for creating Gateway Accounts can be used for both the SAML and X.509 certificate credential sources

  • If the Trusted Gateway A in the identity bridging configuration is not an actual SecureSpan Gateway, then configure the SecureSpan Bridge using the Ad-Hoc Identity Bridging method below. This method for creating a Gateway Account can only be used for the X.509 certificate credential source.

 

For more information about the SAML and X.509 certificate credential source workflows, see Identity Bridging Workflow Using SAML and Identity Bridging Workflow Using an X.509 Certificate, respectively.

 

 NOTE: The generic "Trusted Gateway A" and "Federated Gateway B" references are used in the identity bridging examples, workflows, and instructions. The Trusted Gateway A is the certificate authority (CA) that issues and manages security credentials and that is responsible for authentication. The Federated Gateway B is the Web service provider that is responsible for authorization.

Full Identity Bridging Method

 NOTE: You can use the following method when configuring identity bridging using either the SAML or X.509 certificate credential source.

The following items are required to configure the Gateway Accounts using the Full Identity Bridging method in the SecureSpan™ Bridge:

  • To configure the Trusted Gateway Account, you must have:

    • The host name for the Trusted Gateway A

    • SAML Credential Source: The user name and password for the Internal Identity Provider (IIP) user configured in the Trusted Gateway A and mirrored as a federated user for the Federated Identity Provider in the Federated Gateway B. See Identity Bridging Workflow Using SAML for more information

    • X.509 Credential Source: If using the Full Identity Bridging method in an configuration that uses the X.509 credential source, then you must create a shared federated user between the Trusted Gateway A and Federated Gateway B. To do so, follow the instructions outlined in Identity Bridging Workflow Using SAML before you configure the SecureSpan Bridge. Perform Steps #2-4 to create the IIP user in the Trusted Gateway A, then perform Step #7 to create the shared federated user in the Federated Gateway B. When completed, return to this section to configure the SecureSpan Bridge.

  • To configure the Federated Gateway proxy, you must have:

    • The host name for the Federated Gateway B

    • The previously configured Trusted Gateway proxy entry, uniquely identified by its proxy ID, host name, and user name.

 

In the SecureSpan Bridge, Gateway Accounts are added using the Create Gateway Account form:

 

  1. Add a new Trusted Gateway Account as outlined in Create Trusted Gateway Accounts.

  2. The Gateway Account Properties form appears. In the Gateway Host Name field in the [General] tab, enter the host name and domain of the Trusted Gateway A. For example: “trustedmachinename.domain.com”.

  3. Click the [Identity] tab. Enter the user name of the shared federated user into the User Name field. For example, "user A".

 NOTE: The shared federated user is the Internal Identity Provider user configured in the Trusted Gateway A and mirrored as a federated user for the Federated Identity Provider in the Federated Gateway B.

  1. Enter the password for the shared federated user into the Password field. The password entered in the Password field will remain as long as the SecureSpan Bridge is open on your system. To configure the SecureSpan Bridge to save the password in the Password field after a shutdown, check the optional Save Password check box.

 NOTE: The password for the shared federated user was configured in the Internal Identity Provider user record in the Trusted Gateway A. Since one or more client certificates are used in an identity bridging configuration, do not check the "Use Credentials from Client (HTTP Basic Authentication)" check box.

  1. Click [OK]. The Trusted Gateway Account is added to the Accounts List. Note the Trusted Gateway Account ID in the Proxy column, such as "gateway2".

  1. Add a new Federated Gateway Account as outlined in Create Federated Gateway Accounts. In the Create Gateway Account form:

    • Select the Obtain Security Tokens from a Previously-Created Trusted Gateway option

    • Select the Trusted Gateway Account ID for the configured Trusted Gateway Account from the drop-down list.

  1. Click [Create]. The Gateway Account Properties form appears. In the [General] tab of the Gateway Account Properties form, enter the URL of the Federated Gateway B into the Gateway Host Name field. For example: “federatedmachinename.domain.com”.

  2. Click [OK]. The Federated Gateway Account is added to the Accounts List. You can now use the Federated Gateway Account to consume the shared Web service as outlined in Consume Shared Web Services.

 

Ad-Hoc Identity Bridging Method

 NOTE: You can only use the following method when configuring identity bridging with the X.509 certificate credential source.

The following items are required to configure the Gateway Account using the Ad-Hoc Identity Bridging method in the SecureSpan™ Bridge:

  • The host name for the Federated Gateway B

  • Access to a pre-existing private key PKCS#12 keystore certificate (such as "cert.p12"). The certificate can be self-signed or signed by a CA trusted by the Federated Gateway B. You must also have the PKCS#12 file password.

 

In the SecureSpan Bridge, Gateway Accounts are added using the Create Gateway Account form:

 

  1. Add a new Trusted Gateway Account as outlined in Create Trusted Gateway Accounts.

  2. The Gateway Account Properties form appears. In the Gateway Host Name field in the [General] tab, enter the URL of the Federated Gateway B. For example: https://federatedmachinename.domain.com:8443.

  3. Click the [Identity] tab. Ensure that the User Name and Password fields are empty. Click [View Client Certificate].

 NOTE: Leave the User Name and Password fields empty. The Gateway Account Properties form will automatically populate these values after the PKCS#12 keystore certificate has been imported below.

  1. The Client Certificate Not Found form appears. Click [Import Client Certificate].

 NOTE: You must have access to a pre-existing private key PKCS#12 keystore certificate (such as "cert.p12"). The certificate can be self-signed or signed by a trusted CA. You must also have the PKCS#12 file password.

  1. The Enter New Password form appears. Enter a new password for the SecureSpan Gateway into the Password field. Repeat in the Verify Password field.

  2. Click [OK]. The Select Client Certificate form appears.

  3. Navigate to the location of the certificate. Select the file, and then click [Import Certificate].

  4. The Enter Password form appears. Enter the PKCS#12 file password into the field. Click [OK].

  5. One of two messages will appear:

    • If the password is correct, an import confirmation message appears. Click [OK]. Proceed to Step #10 below

    • If the password is incorrect, an error message will appear. Click [OK], and then repeat Steps #4-8 above. When the certificate has been imported successfully, proceed to Step #10 below.

  1. The User Name and Password fields in the [Identity] tab of the Gateway Account Properties form are automatically populated with the corresponding values from the imported certificate. The password entered in the Password field will remain as long as the SecureSpan Bridge is open on your system. To configure the SecureSpan Bridge to save the password in the Password field after a shutdown, check the optional Save Password check box.

  2. Click [OK]. The Trusted Gateway Account is added to the Accounts List. You can now use the Trusted Gateway Account to consume the shared Web service as outlined in Consume Shared Web Services.
